# v0.1 数据库设计文档

## 1. 文档概述

### 1.1 版本范围

本文档描述 **v0.1 MVP 版本**的数据存储设计。

**v0.1 存储策略**：使用**纯内存存储**，不依赖 SQLite 或任何持久化机制。应用重启后历史记录清空。

SQLite 持久化方案将在 **v0.2** 版本引入，届时数据库设计文档将基于本文档的数据结构定义进行扩展。

### 1.2 设计原则

- 内存结构的设计需与 v0.2 SQLite Schema 保持一致性，便于后续迁移
- 使用 Rust 标准结构体定义，字段名与未来数据库字段名对齐（snake_case）
- 最大记录数：20 条（v0.1 固定，v0.2 起可配置）

## 2. 内存数据结构

### 2.1 主记录结构（ClipboardRecord）

对应未来 SQLite 表 `clipboard_items`（v0.2 引入）。

```rust
/// 粘贴板记录内存结构
/// 字段设计与 v0.2 SQLite clipboard_items 表对齐
#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct ClipboardRecord {
    /// 记录唯一 ID（内存自增，v0.2 改为 SQLite AUTOINCREMENT）
    pub id: u64,

    /// 内容类型（v0.1 固定为 "text"）
    /// 对应未来 TEXT NOT NULL，枚举值：'text' | 'images' | 'files'
    pub content_type: String,

    /// 纯文本内容（v0.1 必填）
    /// 对应未来 TEXT 字段，可为 NULL（非文本记录时）
    pub text_content: String,

    /// 富文本内容 HTML/RTF（v0.1 暂不使用，预留字段）
    /// 对应未来 TEXT 字段，可为 NULL
    #[serde(skip_serializing_if = "Option::is_none")]
    pub rich_content: Option<String>,

    /// 来源应用名称（v0.1 暂不使用，预留字段）
    /// 对应未来 TEXT 字段，可为 NULL
    #[serde(skip_serializing_if = "Option::is_none")]
    pub source_app: Option<String>,

    /// 捕获时间戳（Unix 毫秒）
    /// 对应未来 INTEGER NOT NULL
    pub created_at: i64,
}
```

### 2.2 内存存储容器（InMemoryStore）

```rust
/// 内存存储容器
/// 线程安全，通过 Arc<RwLock<InMemoryStore>> 共享
pub struct InMemoryStore {
    /// 记录列表，索引 0 为最新记录
    records: Vec<ClipboardRecord>,

    /// ID 自增计数器
    next_id: u64,

    /// 最大记录数（v0.1 固定 20）
    max_records: usize,
}

impl InMemoryStore {
    pub fn new(max_records: usize) -> Self;

    /// 插入新记录，超限时淘汰最旧记录
    /// 返回新记录 ID
    pub fn insert(&mut self, text_content: String, created_at: i64) -> u64;

    /// 获取最近 N 条记录（按时间倒序，索引 0 最新）
    pub fn get_recent(&self, limit: usize) -> Vec<ClipboardRecord>;

    /// 根据 ID 获取单条记录
    pub fn get_by_id(&self, id: u64) -> Option<ClipboardRecord>;

    /// 删除指定 ID 的记录
    /// 返回是否删除成功
    pub fn delete(&mut self, id: u64) -> bool;

    /// 获取当前记录总数
    pub fn count(&self) -> usize;

    /// 获取最新一条记录的文本内容（用于去重判断）
    pub fn latest_text(&self) -> Option<&str>;
}
```

## 3. 数据约束与业务规则

### 3.1 数据约束

| 字段 | 约束 | v0.1 规则 |
|------|------|-----------|
| `id` | 正整数，列表内唯一 | 从 1 开始自增，单次会话内不重复 |
| `content_type` | 非空字符串 | v0.1 固定为 `"text"` |
| `text_content` | 非空字符串 | 长度 ≥ 1 字符 |
| `created_at` | Unix 毫秒时间戳 | 使用系统当前时间，单调递增 |
| 记录总数 | ≤ max_records | v0.1 固定上限为 20 |

### 3.2 去重规则

```
新内容是否与历史列表中的任意记录 text_content 完全一致？
    ├── YES，且命中最新记录 → 跳过，不插入
    ├── YES，但命中非最新记录 → 复用原记录并移动到索引 0
    └── NO → 正常插入
```

- 去重比较整个历史列表；命中旧记录时不生成新 ID
- 比较方式：字符串精确匹配，区分大小写

### 3.3 淘汰规则（LRU-like）

```
插入新记录后，列表长度 > max_records？
    ├── YES → 删除索引最大的记录（最旧记录），触发 RecordEvicted 事件
    └── NO  → 无操作
```

## 4. 数据序列化格式

### 4.1 IPC 传输格式（JSON）

后端通过 Tauri IPC 向前端传输记录数据，使用 JSON 序列化。

**单条记录 JSON 示例**：

```json
{
  "id": 5,
  "content_type": "text",
  "text_content": "Hello, World! This is a clipboard record.",
  "created_at": 1709123456789
}
```

**记录列表 JSON 示例**（get_records 返回值）：

```json
[
  {
    "id": 5,
    "content_type": "text",
    "text_content": "最新复制的内容",
    "created_at": 1709123456789
  },
  {
    "id": 4,
    "content_type": "text",
    "text_content": "上一条复制的内容",
    "created_at": 1709123400000
  }
]
```

**字段命名规范**：

| 规范 | 说明 |
|------|------|
| snake_case | 所有 JSON 字段名使用 snake_case（Rust serde 默认） |
| 时间戳 | 统一使用 Unix 毫秒整数（`i64`） |
| 枚举值 | 使用小写字符串（`"text"`，非 `"Text"` 或 `"TEXT"`） |
| 可选字段 | 值为 `null` 的字段在 v0.1 中不序列化（`skip_serializing_if`） |

## 5. 内存存储操作性能预期

| 操作 | 时间复杂度 | 预期耗时（20 条记录） |
|------|-----------|----------------------|
| 插入新记录 | O(1) 均摊 | < 1ms |
| 获取最近 N 条 | O(N) | < 1ms |
| 根据 ID 查询 | O(N) 线性扫描 | < 1ms |
| 删除指定 ID | O(N) 线性扫描 + 移除 | < 1ms |
| 去重检查（最新记录比较） | O(1) | < 0.1ms |

v0.1 最大 20 条记录，所有操作均在微秒级完成，无性能隐患。

## 6. v0.2 迁移预告

v0.2 将引入 SQLite 持久化，数据库表结构设计如下（供参考，不在 v0.1 实现）：

```sql
-- 主记录表（v0.2 引入）
CREATE TABLE clipboard_items (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    content_type TEXT    NOT NULL,          -- 'text' | 'images' | 'files'
    text_content TEXT,
    rich_content TEXT,
    source_app   TEXT,
    created_at   INTEGER NOT NULL,          -- Unix 毫秒时间戳
    is_favorite  INTEGER NOT NULL DEFAULT 0
);

CREATE INDEX idx_items_created_at ON clipboard_items(created_at DESC);
```

v0.1 的内存结构字段与上表字段**一一对应**，确保 v0.2 迁移时无需重构领域模型，只需将 `InMemoryStore` 替换为 `SqliteStorage`（均实现同一 `ClipboardRecordRepository` trait）。

## 7. 错误处理

| 错误场景 | 处理方式 |
|----------|----------|
| 插入空文本（text_content 为空字符串） | 后端校验并返回 `Err`，不插入 |
| 查询不存在的 ID | 返回 `None`，上层决定是否报错 |
| 删除不存在的 ID | 返回 `false`，不抛出错误 |
| 并发读写冲突 | 通过 `RwLock` 保证，写操作独占锁，读操作共享锁 |
