# v0.3 数据库设计文档

## 1. 文档概述

### 1.1 版本范围

`v0.3` 的重点是系统托盘、开机自启动、暂停监听、多显示器与交互增强，而不是新增新的历史记录类型。因此：
- **SQLite 主业务表结构保持与 v0.2 一致**
- 系统集成相关偏好继续存放在 `config.json`
- macOS 自启动通过 Launch Agent plist 落地，不写入数据库

### 1.2 设计原则

- 历史数据仍由 SQLite + 本地文件系统承载
- 运行态控制（是否暂停监听）只存在于内存，不持久化到数据库
- 用户偏好（是否开机自启动）写入 `config.json`
- 清空历史必须保证数据库与图片资源文件双向一致

## 2. 存储布局总览

### 2.1 运行时目录

以 macOS 为例：

```text
~/Library/Application Support/clipboard-record-manager/
├── clipboard.db
├── config.json
├── images/
│   ├── original/
│   └── thumbs/
└── logs/

~/Library/LaunchAgents/
└── com.robin.clipboard-record-manager.plist
```

### 2.2 存储分工

| 介质 | 内容 | 说明 |
|------|------|------|
| `clipboard.db` | 历史记录主数据 | 沿用 v0.2 三表结构 |
| `images/original/` | 图片原图 | 再次粘贴图片时使用 |
| `images/thumbs/` | 图片缩略图 | 主面板预览 |
| `config.json` | 用户偏好 | 包含开机自启动等系统偏好 |
| Launch Agent plist | macOS 登录启动配置 | 属于系统集成产物，不归 SQLite 管理 |

## 3. SQLite 业务模型

### 3.1 版本结论

`v0.3` **不新增业务表，不修改已有字段，不提升 `PRAGMA user_version`**。

当前业务 Schema 仍维持：
- `clipboard_items`
- `image_assets`
- `file_items`

### 3.2 主表 `clipboard_items`

继续沿用 `v0.2`：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER | 主键 |
| `content_type` | TEXT | `text` / `image` / `files` |
| `content_hash` | TEXT | 去重哈希 |
| `text_content` | TEXT | 文本正文 |
| `rich_content` | TEXT | 富文本内容 |
| `preview_text` | TEXT | 卡片摘要 |
| `search_text` | TEXT | 预留搜索字段 |
| `source_app` | TEXT | 来源应用 |
| `file_count` | INTEGER | 文件数 |
| `payload_bytes` | INTEGER | 原始负载大小估算 |
| `created_at` | INTEGER | 首次创建时间 |
| `last_used_at` | INTEGER | 最近被复制或粘贴时间 |

### 3.3 子表与索引

`image_assets`、`file_items` 及其索引继续沿用 `v0.2`，不做 DDL 变更。

这样做的原因：
- `v0.3` 没有新增内容类型
- 清空历史、托盘状态、多显示器定位都不需要新增结构化业务字段
- 保持升级路径简单，减少数据库迁移带来的风险

## 4. 配置存储设计（非 SQLite）

### 4.1 `config.json` 结构

建议 `v0.3` 在 `v0.2` 扁平结构上新增 `launch_at_login`：

```json
{
  "max_text_records": 200,
  "max_image_records": 50,
  "max_file_records": 100,
  "toggle_shortcut": "Shift+Command+V",
  "launch_at_login": true
}
```

### 4.2 配置字段说明

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `max_text_records` | number | `200` | 文本记录上限 |
| `max_image_records` | number | `50` | 图片记录上限 |
| `max_file_records` | number | `100` | 文件记录上限 |
| `toggle_shortcut` | string | `Shift+Command+V` | 呼出主面板快捷键 |
| `launch_at_login` | boolean | `true` | 是否登录启动 |

### 4.3 不落盘的运行态

以下状态**不建议持久化**：
- 当前监听是否暂停
- 当前主面板是否显示
- 当前活动显示器 ID

原因：
- 这些状态属于瞬时运行态
- 跨启动恢复价值低，反而增加脏状态风险

## 5. 清空历史设计

### 5.1 清理范围

用户确认“清空历史”时，系统需要一次性清理：
- `clipboard_items`
- `image_assets`
- `file_items`
- `images/original/*`
- `images/thumbs/*`

**不清理**：
- `config.json`
- 日志目录 `logs/`
- Launch Agent plist

### 5.2 事务与文件一致性策略

建议采用“两阶段语义”：
1. 先读取待删除图片资源路径清单
2. 开启数据库事务删除三张表数据
3. 提交数据库事务后，再删除磁盘图片资源
4. 若图片资源删除失败，记录日志并触发后续孤儿清理任务

这样可以保证：
- 数据库状态优先一致
- 文件系统失败不会回滚已经成功完成的业务清空动作

---

## 6. 迁移策略

### 6.1 Schema 版本

`v0.3` 建议保持：

```rust
pub const CURRENT_SCHEMA_VERSION: u32 = 2;
```

### 6.2 配置兼容迁移

若旧版 `config.json` 不包含 `launch_at_login` 字段：
- 启动时自动回填默认值 `true`
- 回写完整配置文件

### 6.3 自启动文件修复

启动时可做弱校验：
- 若 `config.json.launch_at_login = true` 且 Launch Agent 缺失，则尝试补建
- 若 `config.json.launch_at_login = false` 但 Launch Agent 仍存在，则尝试清理

此逻辑只修复“配置与系统状态不一致”，不修改历史数据库。

---

## 7. 性能与可靠性目标

| 项目 | 目标 |
|------|------|
| `get_records(100)` | 保持 `< 80ms` |
| 清空历史（200 文本 + 50 图片 + 100 文件） | `< 300ms` 数据库阶段完成 |
| 启动读取配置 + 自启动状态 | `< 30ms` |
| 主面板多显示器定位计算 | `< 10ms` |

---

## 8. 与 v0.2 的主要差异

| 项目 | v0.2 | v0.3 |
|------|------|------|
| SQLite Schema | 三表结构 | 不变 |
| 配置项 | 容量上限 + 快捷键 | 新增 `launch_at_login` |
| 系统集成存储 | 无 | 新增 Launch Agent plist |
| 批量清理 | 超限淘汰 / 单条删除 | 新增“清空全部历史” |

结论：`v0.3` 是**系统集成与运行治理升级**，不是**业务数据模型升级**。
