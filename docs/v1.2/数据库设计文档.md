# v1.2 数据库设计文档

## 1. 文档概述

### 1.1 版本范围

`v1.2` 的改动全部集中在**运行态交互、图片预览解析与设置页信息架构**，不涉及新的业务持久化模型。

### 1.2 设计原则

- **不为纯运行态 UI 概念落表**：鼠标选中态、可视槽位编号、当前激活设置分组都只存在于内存
- **不为展示修复强行迁移结构**：图片预览修复优先复用现有 `thumbnail_path / original_path`
- **数据库排序仍是真相源**：可视区域编号只是在既有排序之上的运行态映射

---

## 2. 存储布局总览

### 2.1 版本结论

| 项目 | v1.2 结论 |
|------|-----------|
| SQLite 业务表 | 不新增 |
| `schema_version` | 继续沿用 `2` |
| `config_version` | 继续沿用 `2` |
| 数据迁移脚本 | 不新增 |
| 运行态内存字段 | 新增若干 UI 级派生状态 |

### 2.2 继续沿用的 SQLite 业务模型

- `clipboard_items`
- `image_assets`
- `file_items`

这些表已经足以支撑 `v1.2` 的需求：

- 鼠标单击 / 双击仅改变交互触发方式，不改变记录实体
- 可视槽位编号依赖现有记录顺序，不新增编号列
- 图片预览修复复用既有图片元数据与详情路径，不新增预览资源表

---

## 3. 运行态数据设计

### 3.1 应进入运行态内存、而非数据库的数据

| 数据 | 建议存放位置 | 原因 |
|------|--------------|------|
| `selection_source` | 前端 `ClipboardStore / UIStore` | 只对当前交互会话有效 |
| `visible_slot_start_index` | 前端 `UIStore` | 依赖滚动位置即时变化 |
| `visible_slot_map` | 前端派生数据 | 可由列表与滚动状态随时重算 |
| `is_pasting` | 前端局部状态 | 用于防双击重复提交 |
| `active_settings_section` | 设置窗口局部状态 | 只是窗口导航状态 |
| `preview_source_kind` | 图片卡片局部状态 | 只影响当前组件渲染 |

### 3.2 为什么不新增业务表

- 鼠标交互是 UI 输入方式变化，不是新的业务对象
- 可视槽位编号是瞬时视口映射，持久化后反而会和滚动位置强耦合
- 设置页左 / 右布局与能力独立分组属于信息架构变化，不是配置本身变化
- 图片预览问题属于资源解析与渲染链路问题，不是数据结构缺字段

---

## 4. 配置存储设计（`config.json`）

### 4.1 `v1.2` 结构结论

`v1.2` 不新增配置字段，继续沿用 `v1.0 / v1.1` 的 `config_version=2` 结构：

- `general`
- `history`
- `shortcut`
- `privacy`

### 4.2 为什么本版不加入“设置布局偏好 / 快捷编号模式开关”

- “左侧导航 + 右侧内容”是当前版本统一后的默认布局，不提供用户自定义切换
- “可视区域快捷编号”是为修复当前使用断层而设定的标准行为，不在 `v1.2` 暴露双模式配置
- 若未来确实出现“绝对编号 / 可视编号”共存需求，再在后续版本评估配置化

---

## 5. 查询与排序约束

### 5.1 可视槽位编号来源

- 记录基础顺序继续以 `last_used_at DESC`（以及现有稳定次序）为准
- `visible_slot_start_index` 由当前列表的可视起点决定
- 槽位 `1~9` 映射为：`visible_slot_start_index + 0..8`

示例：

| 当前视口左侧第一个可见卡片绝对索引 | 槽位映射 |
|------------------------------------|----------|
| `0` | `1->0, 2->1, 3->2 ...` |
| `5` | `1->5, 2->6, 3->7 ...` |
| `12` | `1->12, 2->13, 3->14 ...` |

### 5.2 图片预览查询约束

- 列表渲染优先使用摘要中的 `image_meta.thumbnail_path`
- 当缩略图不可用时，仅为**当前可见图片卡片**按需读取详情中的 `image_detail.original_path`
- 不允许为整个长列表一次性加载所有图片详情，避免无谓 IO 与内存占用

---

## 6. 性能与可靠性目标

- 可视槽位重算应控制在当前视口范围内，避免随滚动触发整表级昂贵计算
- 图片预览解析失败时必须平滑降级到占位态，不能出现控制台刷错或破损图片图标
- 双击粘贴只发起一次实际 `paste_record` 请求
- 设置页切换分组时不触发配置层重复读取或多余写入

---

## 7. 与 v1.1 的主要差异

- `v1.1` 把重点放在窗口行为与键盘链路；`v1.2` 聚焦纯运行态 UI 收敛
- `v1.1` 的快捷编号默认为绝对索引；`v1.2` 明确改为视口映射，但该映射仍不落盘
- `v1.2` 对数据库的主要要求不是新增结构，而是继续保证记录顺序、图片路径与详情读取的稳定性
