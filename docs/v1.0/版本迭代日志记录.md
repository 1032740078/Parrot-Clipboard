## 2. 本次变更明细

### 2.1 2026-03-08 `v1.0-db-recovery-summary`

**变更类型**：`feat(persistence)`

**主要变更**：
- 在 `src-tauri/src/persistence/mod.rs` 新增 `MigrationStatus` 与 `PersistenceInitialization`，统一描述本轮启动的迁移 / 恢复结果
- 在 `src-tauri/src/persistence/sqlite.rs` 为初始化流程补充 `migrated` 标记，并记录 `sqlite database initialized` 的迁移状态
- 启动阶段会输出 `sqlite startup summary`，包含当前 schema、是否迁移、是否损坏恢复、检查时间与备份路径
- 在 `src-tauri/src/state.rs`、`src-tauri/src/lib.rs` 中把迁移摘要注入全局状态，供后续诊断接口直接复用

**产出结果**：
- `SQLite` 已不只是“能恢复”，还会在启动后保留结构化恢复摘要
- 后续 `ReleaseInfo / DiagnosticsSnapshot` 可以直接读取当前会话迁移状态，而不必重复扫描日志
- 自动化已覆盖“干净初始化”和“损坏恢复重建”两类发布高风险场景

**验证结果**：
- `cargo test --manifest-path src-tauri/Cargo.toml initialize_with_recovery -- --nocapture` ✅
- `cargo test --manifest-path src-tauri/Cargo.toml initialize_database_creates_tables_indexes_and_schema_version -- --nocapture` ✅

### 2.2 2026-03-08 `v1.0-log-bridge`

**变更类型**：`feat(logging)`

**主要变更**：
- 启用 `src-tauri/src/logging.rs` 的统一日志初始化，支持控制台、滚动文件与 `panic` 捕获
- 通过 `src-tauri/src/ipc/commands.rs` 暴露 `write_client_log`、`get_log_directory`，供前端桥接调用
- 在 `src/api/logger.ts`、`src/api/commands.ts`、`src/api/events.ts`、`src/main.tsx` 等前端关键链路接入结构化日志上报
- 托盘菜单提供“打开日志目录”动作，便于人工验收与问题排查

**产出结果**：
- 前后端错误已可落到同一轮会话的本地日志目录中
- 前端日志会自动做上下文清洗与消息截断，降低敏感信息与超长日志风险
- 日志目录可以通过 IPC 查询，也可以从托盘直接打开

**验证结果**：
- `pnpm test src/__tests__/api/logger.test.ts src/__tests__/api/commands.test.ts src/__tests__/api/events.test.ts` ✅

## 3. 当前阶段结论

### 已锁定事项

- `v1.0` 不是新业务功能扩展版本，而是正式发布收口版本。
- 当前优先级以 `P0` 的迁移恢复、日志诊断、权限引导、性能稳定性为主。
- `v1.0` 保持 `v0.5` 的设置结构与三表业务模型稳定，不再新增业务表。

### 下一步建议

1. 按 `docs/v1.0/冲刺任务文档.md` 先完成 `TASK-01-*` 发布底座任务。
2. 优先建立数据库迁移 / 恢复、日志目录与诊断快照的后端能力。
3. 再进入关于页、权限引导、虚拟滚动与自动化验收补齐阶段。
