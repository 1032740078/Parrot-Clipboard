### 2.1 2026-03-08 `v1.0-image-cache-query-optimization`

**变更类型**：`feat(image)`

**主要变更**：
- `src-tauri/src/image/storage.rs` 新增原图 `LRU` 缓存，采用 `32` 条目上限与 `64MB` 字节预算双阈值控制，并在缓存命中时跳过磁盘读取与解码
- `remove_assets` 删除原图 / 缩略图时会同步驱逐对应缓存项，避免图片文件已清理但内存仍保留旧资源
- `src-tauri/src/persistence/sqlite.rs` 的 `SUMMARY_SELECT_SQL` 已改为在 SQL 层直接计算 `text_char_count / text_line_count`，不再把完整 `text_content` 带回最近历史摘要查询
- `src-tauri/src/persistence/row_mapper.rs` 已改为消费新的摘要列结构，并补上空文本 `TextMeta` 的兼容处理
- 新增 `LRU` 命中 / 淘汰 / 驱逐与空文本摘要元数据测试，收口本任务的性能与一致性风险

**产出结果**：
- 高频图片粘贴场景下，原图恢复链路已经具备复用内存副本的能力，降低重复解码开销
- 最近历史摘要查询不再无意义搬运大段正文，为主面板 `200+` 记录场景继续降低 IO 与映射成本
- 图片文件删除与缓存生命周期已经对齐，为后续 `TASK-03-3` 的孤立文件清理打下基础

**验证结果**：

## 3. 当前阶段结论

### 已锁定事项

- `v1.0` 不是新业务功能扩展版本，而是正式发布收口版本。
- 当前优先级以 `P0` 的迁移恢复、日志诊断、权限引导、性能稳定性为主。
- `v1.0` 保持 `v0.5` 的设置结构与三表业务模型稳定，不再新增业务表。

### 下一步建议

1. 按 `docs/v1.0/冲刺任务文档.md` 先完成 `TASK-01-*` 发布底座任务。
2. 优先建立数据库迁移 / 恢复、日志目录与诊断快照的后端能力。
3. 再进入关于页、权限引导、虚拟滚动与自动化验收补齐阶段。
