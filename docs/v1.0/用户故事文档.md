# v1.0 用户故事文档

## 1. 文档概述

### 1.1 版本范围

本文档仅描述 **v1.0 Release 版本** 的用户故事。

`v1.0` 以 `v0.5` 已完成的三平台主流程、设置中心、快捷键、隐私黑名单与能力降级为前提，当前版本只聚焦以下新增或强化能力：

- 正式发布级稳定性：数据库迁移、损坏数据库恢复、异常退出后重启可继续工作
- 性能优化：主面板虚拟滚动、图片原图 `LRU` 缓存、查询与清理性能收口
- 权限与引导：`macOS` 辅助功能权限缺失时给出明确引导
- 版本与诊断：关于页面、版本信息、日志目录、应用内手动检查更新
- 存储治理：孤立图片文件定期扫描与清理、数据库与图片资产一致性保障

以下能力 **不属于 `v1.0` 当前版本范围**：

- 全文搜索、类型筛选、收藏、标签、置顶
- 云同步、账号体系、导入导出、插件系统
- OCR、智能分类、命令行接口、脚本扩展
- 复杂工作流自动化、跨设备协作能力

### 1.2 角色定义

| 角色 | 描述 | 主要诉求 |
|------|------|----------|
| 日常办公用户 | 长时间依赖快捷键与主面板完成复制粘贴的用户 | 响应快、不卡顿、重启后仍可继续使用 |
| 隐私敏感用户 | 经常复制密码、验证码、客户信息的用户 | 数据不误入库、异常时有清晰提示 |
| 大历史重度用户 | 历史记录多、图片记录多的用户 | 长列表依然流畅，图片不会拖垮内存 |
| 支持与维护人员 | 负责定位用户问题、收集日志、辅助升级的人员 | 能快速拿到版本信息、日志目录、诊断上下文 |

## 3. 用户故事详细说明

### E01 - 发布可靠性

#### US-01-01：升级到新版本时历史数据自动迁移

**作为** 已在使用 `v0.5` 的老用户  
**我希望** 升级到 `v1.0` 后数据库迁移能在启动时自动完成  
**以便** 不需要手工处理数据库文件，也不会因为版本升级丢失历史记录。

**验收要点**：
- 启动时自动检测数据库 `schema_version`
- 若存在待执行迁移，按顺序执行并保证幂等
- 迁移成功后历史记录仍可正常浏览、粘贴、删除
- 若数据库损坏，先备份后重建，且给出可诊断线索

#### US-01-02：异常退出后再次启动可恢复工作

**作为** 长期开机运行工具的用户  
**我希望** 应用异常退出后重新启动仍能恢复到可工作状态  
**以便** 不会因为一次异常导致整个工具不可用。

**验收要点**：
- 损坏数据库会被自动隔离备份，不阻塞启动
- 重建后的应用仍能继续采集新记录
- 日志中保留异常与恢复过程，但不记录敏感正文
- 托盘、主面板、设置窗口的基本入口仍可恢复

### E02 - 权限与平台引导

#### US-02-01：macOS 权限不足时得到明确引导

**作为** 初次安装 `macOS` 版本的用户  
**我希望** 当辅助功能权限不足时看到明确步骤说明  
**以便** 能完成快捷键呼出与自动粘贴所需授权。

**验收要点**：
- 仅在 `macOS` 且权限缺失时展示引导入口
- 引导内容包含“为什么需要此权限”“如何前往系统设置”“授权后如何重试”
- 未授权时禁止进入无反馈状态，必须提供替代说明
- 授权完成后无需重装应用即可重新检测

#### US-02-02：能力受限时仍可浏览历史

**作为** 运行在受限环境中的用户  
**我希望** 即使快捷键或活动应用识别不可用，仍可打开主面板浏览已有历史  
**以便** 降级只影响受限能力，不影响基本可用性。

**验收要点**：
- 降级能力必须有显式文案说明
- 不可用能力仅禁用对应操作，不影响记录列表渲染
- 历史浏览、删除、清空等非受限能力仍可工作

### E03 - 版本与诊断

#### US-03-01：查看版本、平台与许可证信息

**作为** 需要确认安装状态的用户  
**我希望** 在关于页面看到应用版本、平台信息、数据库版本与开源协议入口  
**以便** 快速判断当前安装状态并向他人反馈。

**验收要点**：
- 关于页展示 `app_version`、`platform`、`session_type`、`schema_version`
- 提供开源协议或许可证列表入口
- 页面信息可复制或方便截图反馈

#### US-03-02：在应用内手动检查更新

**作为** 希望保持最新稳定版本的用户  
**我希望** 在应用内手动执行“检查更新”  
**以便** 不必每次去发布页手工比对版本。

**验收要点**：
- 可明确区分“已有新版本”“当前已最新”“检查失败”三种结果
- 检查过程有加载态，结束后有可读反馈
- 检查失败时不影响主面板与设置窗口继续使用

#### US-03-03：遇到问题时能快速定位日志目录

**作为** 需要排查问题的用户或维护人员  
**我希望** 能从应用中拿到日志目录与最近一次错误线索  
**以便** 更快提交问题和定位故障。

**验收要点**：
- 支持显示本地日志目录路径
- 日志目录不存在时可自动创建或给出提示
- 前端错误、后端错误与 panic 都能落到本地日志文件

### E04 - 性能与大历史可用性

#### US-04-01：长历史列表仍然流畅滚动与切换

**作为** 有大量历史记录的用户  
**我希望** 主面板在几十到上百条记录下仍保持平滑滚动和键盘切换  
**以便** 不因列表过长而降低操作效率。

**验收要点**：
- 主面板仅渲染可见区与必要缓冲区卡片
- 键盘左右切换不会因虚拟滚动丢失焦点
- 记录高亮、快速选择与删除行为保持一致

#### US-04-02：图片记录不会造成持续内存攀升

**作为** 经常复制截图的用户  
**我希望** 应用只缓存必要的图片资源，并在空闲时释放原图缓存  
**以便** 长时间运行后内存仍保持稳定。

**验收要点**：
- 原图缓存采用 `LRU` 策略并限制上限
- 卡片默认使用缩略图，只有真正粘贴或预览时才加载原图
- 长时间运行后内存增长保持在预算范围内

### E05 - 数据一致性与清理

#### US-05-01：孤立图片文件会被自动识别并清理

**作为** 长期使用图片复制功能的用户  
**我希望** 数据库已不存在引用的图片文件能被周期性清理  
**以便** 避免磁盘空间被无效文件持续占用。

**验收要点**：
- 清理前先通过数据库引用扫描判断是否孤立
- 只删除未被引用的原图和缩略图文件
- 清理结果写入日志并返回统计信息

#### US-05-02：清空历史后数据库与文件资产保持一致

**作为** 希望重置历史的用户  
**我希望** 清空历史时不仅删除数据库记录，也一并清理图片资源  
**以便** 不会留下数据库为空但磁盘仍被占用的残留文件。

**验收要点**：
- 清空历史后 `clipboard_items`、`image_assets`、`file_items` 与图片目录状态一致
- 清空操作可审计，日志中记录删除数量与资源数量
- 清空失败时不出现部分成功但界面无提示的情况

## 5. 排除范围说明

- `v1.0` 不新增搜索结果页、收藏页、标签页等全新业务页面。
- `v1.0` 不引入账号、同步、远程 API 或服务端存储。
- `v1.0` 的更新能力仅定义为 **检查与引导安装**，不要求在当前版本中完成复杂的静默升级流程。
- `v1.0` 继续沿用 `v0.5` 的设置领域模型，不在本版本引入新的复杂配置分组。
