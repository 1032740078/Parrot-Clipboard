# v1.0 UX 设计文档

## 1. 文档概述

### 1.1 版本范围

本文档仅描述 `v1.0 Release` 当前版本的体验设计重点。

### 1.2 设计原则（v1.0 强调）

1. **稳于新增**：正式发布阶段优先提升稳定感，而不是堆叠新功能。
2. **失败可解释**：权限不足、更新失败、数据库恢复都必须有清晰反馈。
3. **性能可感知**：虚拟滚动和缓存优化不仅要快，还要让用户感知到稳定和顺滑。
4. **诊断可达**：用户遇到问题时，能在应用内找到日志、版本、平台和恢复线索。

---

## 2. 主面板性能 UX 规范

### 2.1 虚拟滚动体验

- 虚拟滚动应对用户透明，不能改变原有卡片浏览心智。
- 高亮记录切换必须和视图滚动保持同步。
- 大列表滚动过程中避免出现大面积白屏或跳动。

### 2.2 图片缓存体验

- 默认仅加载缩略图，保证主面板始终轻量。
- 需要原图时再按需装载，并优先复用缓存。
- 缓存淘汰应静默进行，不向用户暴露技术性细节。

### 2.3 反馈要求

- 删除、粘贴、清空、重试等操作继续使用统一反馈体系。
- 若由于权限或能力限制导致操作失败，反馈文案必须指向可执行的下一步。

---

## 3. 权限引导 UX

### 3.1 展示原则

- 只在真正需要时展示，不对非 `macOS` 用户造成干扰。
- 先讲“为什么需要”，再给“怎么去做”，最后给“做完如何回来重试”。
- 避免只给系统术语，不给用户可执行步骤。

### 3.2 典型文案结构

1. 当前能力受限的原因
2. 权限用途说明
3. 打开系统设置的入口
4. 授权完成后的重试说明

### 3.3 交互要求

- “稍后处理”允许用户继续浏览历史。
- “打开系统设置”失败时提供手工路径说明。
- “重新检测”结果应立刻反馈，不让用户猜测是否成功。

---

## 4. 关于与更新 UX

### 4.1 信息架构

关于页优先展示以下顺序：

1. 当前版本与平台信息
2. 检查更新操作
3. 诊断与日志目录
4. 法务与许可证入口

### 4.2 更新检查体验

- 默认是用户主动触发，不在 `v1.0` 中做侵入式强提醒。
- 检查中必须有明确 loading 态。
- 结果分为“发现更新”“已是最新”“检查失败”三种，不混用文案。

### 4.3 诊断入口体验

- 日志目录路径要可见，也要可打开。
- 版本、数据库版本、恢复状态适合集中展示，方便截图反馈。
- 不把过多技术指标塞给普通用户；高级信息可折叠展示。

---

## 5. 恢复与清理 UX

### 5.1 数据库恢复体验

- 若应用已自动恢复数据库，应以“已恢复可继续使用”为第一信息。
- 技术细节放到日志或诊断区，而不是主流程首屏大量干扰。
- 若恢复失败，才进入阻断性错误提示。

### 5.2 清理体验

- 孤立文件清理默认后台低打扰执行。
- 若提供手动清理入口，需要明确“只清理无引用文件”。
- 清理结果以统计信息呈现，避免展示过多底层路径细节。

---

## 6. 反馈设计

### 6.1 操作反馈矩阵

| 场景 | 反馈形式 | 说明 |
|------|----------|------|
| 轻量成功 | Toast | 如保存成功、检查完成 |
| 可重试失败 | Toast + CTA | 如更新检查失败、日志目录打开失败 |
| 权限受限 | 提示条 / 弹窗 | 引导型反馈，优先说明下一步 |
| 阻断失败 | 模态 / 独立错误页 | 如启动恢复失败 |

### 6.2 Toast 规范

- 成功：简短、明确、不重复技术名词
- 失败：先说结果，再给动作建议
- 降级：强调“仍可继续使用哪些能力”

---

## 7. 可用性指标

| 指标 | 目标 |
|------|------|
| 主面板可交互时间 | `< 200ms` |
| 快捷键响应 | `< 100ms` |
| 日志目录定位 | 用户 `2` 步内可到达 |
| 权限引导完成闭环 | 用户可在一次引导中理解并完成操作 |
| 长列表滚动稳定性 | 不出现明显白屏、跳动、焦点丢失 |

---

## 8. 平台特定 UX 规范

### 8.1 macOS

- 权限引导必须贴近系统术语，但避免纯技术描述。
- 视觉和交互尽量轻量，不抢占主面板效率心智。

### 8.2 Windows

- 关于页中的更新与日志入口要适应更直白的系统反馈习惯。
- 避免使用只适用于 `macOS` 的权限术语。

### 8.3 Linux

- 继续保留对 `X11 / Wayland` 差异的明确说明。
- 当能力受限时，优先说明“为什么不可用”和“仍可做什么”。
