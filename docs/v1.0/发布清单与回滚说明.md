# v1.0 发布清单与回滚说明

> 更新时间：`2026-03-08`
> 适用版本：`v1.0.0`
> 目标：在自动化回归通过后，完成三平台打包、签名、校验与回滚准备

## 1. 发布产物范围

| 平台 | 产物 | 签名策略 | 备注 |
|------|------|----------|------|
| macOS | `.app` + `.dmg` | `.app` 由 Tauri 完成签名 / notarization，`.dmg` 由 `package-macos-dmg.sh` 进行无界面封装 | `workflow: release.yml` 在 `macos-latest` 执行 |
| Windows | `.exe`、`.msi` | `signtool` + 证书指纹签名 | 通过 `scripts/sign-windows.ps1` 执行 |
| Linux | `.AppImage`、`.deb`、`.rpm` | 不做代码签名，发布 `SHA256SUMS.txt` | 通过校验和保证分发完整性 |

## 2. 必备 GitHub Secrets

| Secret 名称 | 用途 | 平台 |
|-------------|------|------|
| `APPLE_CERTIFICATE` | Base64 编码的 `Developer ID Application` 证书 `.p12` | macOS |
| `APPLE_CERTIFICATE_PASSWORD` | 证书密码 | macOS |
| `APPLE_SIGNING_IDENTITY` | Apple 签名身份，例如 `Developer ID Application: ...` | macOS |
| `APPLE_API_ISSUER` | App Store Connect Issuer ID | macOS |
| `APPLE_API_KEY_ID` | App Store Connect Key ID | macOS |
| `APPLE_API_KEY_CONTENT` | App Store Connect `.p8` 私钥原文 | macOS |
| `WINDOWS_CERTIFICATE` | Base64 编码的 `.pfx` 证书 | Windows |
| `WINDOWS_CERTIFICATE_PASSWORD` | `.pfx` 密码 | Windows |
| `WINDOWS_CERTIFICATE_THUMBPRINT` | 可选，指定签名证书指纹；留空时自动读取导入结果 | Windows |
| `WINDOWS_SIGN_DIGEST_ALGORITHM` | 可选，默认 `sha256` | Windows |
| `WINDOWS_TIMESTAMP_URL` | 可选，默认 `http://timestamp.digicert.com` | Windows |
| `WINDOWS_TIMESTAMP_TSP` | 可选，若签名服务要求 RFC3161 则设置为 `true` | Windows |

## 3. 发布前检查清单

### 3.1 自动化验收

在打 tag 前必须完成以下命令：

```bash
pnpm run release:verify-config
pnpm run release:check
pnpm run release:build:local
```

说明：
- `release:verify-config`：校验版本号、发布 workflow、签名脚本与文档是否齐备
- `release:check`：串行执行类型检查、Lint、覆盖率测试、Rust 单测、Playwright BDD 与发布配置校验
- `release:build:local`：本机先生成 unsigned `.app`，再通过 `scripts/package-macos-dmg.sh` 执行无界面 `dmg` 封装，验证当前平台打包链路可用

### 3.2 版本与内容校对

- 确认 `package.json`、`src-tauri/Cargo.toml`、`src-tauri/tauri.conf.json` 的版本一致并为 `1.0.0`
- 确认 `docs/v1.0/冲刺任务文档.md`、`docs/v1.0/版本迭代日志记录.md` 已同步到最终完成态
- 确认更新源 `CLIPBOARD_UPDATE_FEED_URL` 指向正式发布清单或 GitHub Release 接口
- 确认 `README`、发行说明、已知限制与安装说明已经准备完毕

### 3.3 GitHub Release 流程

1. 在 `main` 分支确认 `CI` 全绿。
2. 打版本 tag，例如：`v1.0.0`。
3. 触发 `.github/workflows/release.yml`：
   - 先执行 `release:check`
   - 再在 `macOS / Windows / Linux` 并行打包
   - 自动收集 `.dmg / .exe / .msi / .AppImage / .deb / .rpm`（macOS 的 `.dmg` 由 `package-macos-dmg.sh` 生成）
   - 生成 `SHA256SUMS.txt`
   - 创建或更新 GitHub Draft Release
4. 人工核对 Draft Release 的版本号、产物数量、校验和与发行说明后再发布。

## 4. 本地打包说明

### 4.1 macOS

```bash
pnpm install --frozen-lockfile
pnpm run release:build:local
```

- 本地默认使用 `--no-sign` 冒烟构建，不要求开发机持有发布证书；DMG 通过 `hdiutil` 无界面封装，避免 Finder AppleScript 在终端 / CI 环境卡住。
- 若需要真机验证签名 / notarization，请在受控环境中导入 Apple 证书并设置对应环境变量后再执行正式构建。

### 4.2 Windows

- 正式签名由 `release.yml` 负责导入 `.pfx` 并通过 `scripts/sign-windows.ps1` 调用 `signtool`。
- 如需本地验证，请先安装 Windows SDK 与有效证书，再设置：
  - `WINDOWS_CERTIFICATE_THUMBPRINT`
  - `WINDOWS_SIGN_DIGEST_ALGORITHM`
  - `WINDOWS_TIMESTAMP_URL`

### 4.3 Linux

- Linux 发布产物不做代码签名。
- 发布时必须附带 `SHA256SUMS.txt`，供用户下载后校验完整性。

## 5. 回滚说明

### 5.1 发布前回滚

若 Draft Release 构建完成后发现问题：

1. 保持 Draft 状态，不要点击发布。
2. 删除或覆盖错误产物。
3. 修复代码后重新打同名 tag 或删除 tag 后重新创建。
4. 重新触发 `release.yml`，确认新的 `SHA256SUMS.txt` 与安装包一致。

### 5.2 已发布版本回滚

若 `v1.0.0` 正式发布后发现阻塞问题：

1. 立即将 GitHub Release 标记为 `pre-release` 或撤回下载入口。
2. 将 `CLIPBOARD_UPDATE_FEED_URL` 指回上一稳定版本的 release / manifest，避免新用户继续获取问题版本。
3. 发布修复版本（如 `v1.0.1`）前，保留上一稳定版本安装包与 `SHA256SUMS.txt`。
4. 在发行说明中明确：
   - 受影响平台
   - 回退方式
   - 是否需要先卸载再安装旧版本
   - 数据目录与数据库是否兼容

### 5.3 用户侧回退原则

- 数据目录默认保留，不应在回退说明中要求用户手工删除数据库。
- 若回退到更旧版本存在配置兼容风险，应优先通过热修复版本解决，而不是直接要求用户长期停留在旧版。
- 若必须人工回退，优先指导用户备份日志目录、`config.json` 与数据库文件后再执行降级安装。

- `release:check` 全部通过。

- GitHub Release Draft 同时包含三平台安装产物与 `SHA256SUMS.txt`。

- 发行说明、已知限制、回滚说明与冲刺文档状态保持一致。
