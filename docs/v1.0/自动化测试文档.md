# v1.0 自动化测试文档

## 1. 文档概述

### 1.1 版本范围

本文档仅描述 `v1.0 Release` 当前版本的自动化测试策略。

`v1.0` 的自动化重点从 `v0.5` 的“功能补齐”转向“发布质量收口”，测试范围重点覆盖：

- 数据库迁移、损坏恢复与清理边界
- 关于页、日志诊断、更新检查
- `macOS` 权限引导与能力降级体验
- 主面板虚拟滚动、图片缓存与长时运行稳定性
- 三平台回归与正式发布前的 CI 收口

### 1.2 测试架构分层

| 层级 | 工具 | 目标 |
|------|------|------|
| 后端单元测试 | `cargo test` | 验证迁移、恢复、日志、清理、平台适配与领域规则 |
| 前端单元测试 | `Vitest + RTL + jsdom` | 验证主面板、关于页、权限引导、状态同步与错误反馈 |
| 后端集成测试 | `cargo test` | 验证 `SQLite + 文件目录 + 服务编排` 联动 |
| BDD / E2E | `Playwright` | 从用户视角验证发布级关键链路 |
| 静态检查 | `ESLint`、`TypeScript`、`cargo fmt`、`clippy` | 保证发布前代码质量稳定 |

### 1.3 覆盖率目标

| 范围 | 目标 |
|------|------|
| 前端覆盖率 | `lines / branches / functions / statements >= 75%` |
| 后端关键模块 | 迁移、恢复、清理、日志、权限、平台能力需覆盖核心分支 |
| BDD 主链路 | `P0` 用户故事必须具备自动化验收场景 |
| 发布回归 | `lint + type-check + test:coverage + cargo test + test:e2e` 全通过 |

## 2. 后端单元测试（Rust / cargo test）

### 2.1 数据库迁移与恢复

覆盖目标：

- `user_version = 0/1/2` 时的迁移分支
- 重复启动时迁移幂等
- 数据库损坏时触发“备份后重建”
- 重建后可重新建表并正常读写

推荐用例：

- `ut_migrate_001`：空库初始化后 `schema_version=2`
- `ut_migrate_002`：旧库升级到最新版本
- `ut_recover_001`：损坏库重命名为 `corrupt-*` 备份并成功重建
- `ut_recover_002`：存在 `-wal/-shm` 时一并备份

### 2.2 日志与诊断

覆盖目标：

- 日志目录解析与创建
- 前端 `write_client_log` 写入成功
- panic hook 安装后可写入 panic 日志
- 诊断快照能汇总日志目录与关键版本信息

推荐用例：

- `ut_log_001`：首次启动自动创建日志目录
- `ut_log_002`：前端 `debug/info/warn/error` 均能落盘
- `ut_log_003`：重复初始化不会重复安装 panic hook

### 2.3 图片资产与清理

覆盖目标：

- 原图保存与缩略图生成
- 孤立原图 / 缩略图扫描
- 清空历史时数据库与图片资源共同删除
- `LRU` 原图缓存淘汰顺序正确

推荐用例：

- `ut_image_001`：缩略图尺寸不超过 `200x200`
- `ut_image_002`：孤立文件扫描结果正确
- `ut_image_003`：`remove_assets` 可删除原图与缩略图
- `ut_cache_001`：缓存超过上限后淘汰最久未使用条目

### 2.4 权限与平台能力

覆盖目标：

- `macOS` 辅助功能权限缺失 / 已授权 / 无法检测三种状态
- `Wayland` 与 `X11` 能力矩阵差异
- 权限缺失时命令层返回结构化错误或降级状态

推荐用例：

- `ut_perm_001`：`macOS` 未授权时返回 `missing`
- `ut_perm_002`：授权完成后状态变为 `granted`
- `ut_caps_001`：Wayland 下全局快捷键为 `unsupported/degraded`

### 2.5 更新检查与发布元信息

覆盖目标：

- 新版本可用、当前最新、网络失败三类结果
- 更新源返回异常数据时的容错
- 关于页读取的 `ReleaseInfo` 字段完整性

推荐用例：

- `ut_update_001`：存在更高版本时返回 `available`
- `ut_update_002`：当前已是最新版本时返回 `latest`
- `ut_update_003`：更新源失败时返回结构化错误

## 3. 前端单元测试（Vitest + RTL）

### 3.1 主面板性能与交互

覆盖重点：

- 虚拟滚动只渲染可见卡片
- 键盘左右切换、数字键快选与虚拟列表共存
- 删除、粘贴、清空确认在虚拟列表下行为稳定
- 大量记录下骨架屏与空态切换正确

### 3.2 关于页与诊断入口

覆盖重点：

- 版本、平台、数据库版本、日志目录展示
- “检查更新”加载态与结果态切换
- 打开日志目录、复制版本信息等操作反馈
- 更新失败不会打断其他页面可用性

### 3.3 权限引导与降级提示

覆盖重点：

- `macOS` 权限缺失时展示引导弹窗 / 提示条
- 非 `macOS` 环境不误展示辅助功能引导
- 降级提示文案与禁用态按钮一致
- 授权后重试可刷新状态

### 3.4 设置与运行态回归

覆盖重点：

- `v0.5` 已有设置页功能不回退
- 主题切换后主面板和关于页视觉令牌一致
- 日志、更新、权限入口不会破坏现有设置流程

## 4. 后端集成测试

### 4.1 启动链路

- 配置加载 → 日志初始化 → 数据库初始化 / 迁移 → 图片目录初始化 → 运行态启动
- 任一环节失败时错误信息可定位、可追踪

### 4.2 数据一致性链路

- 新增图片记录后：数据库、原图、缩略图三者一致
- 删除图片记录后：数据库与文件同步删除
- 清空历史后：表与图片目录清空，配置与日志保留

### 4.3 诊断链路

- 日志目录可读取
- 诊断快照可汇总数据库版本、最近恢复结果、最近清理结果
- 前端日志与后端日志可在同一目录中检索

### 4.4 更新检查链路

- 通过 mock 更新源验证成功 / 失败 / 超时
- 不允许网络失败导致主线程阻塞或窗口卡死

## 5. BDD / E2E 自动化

### 5.1 建议工具链

- `Playwright`
- `Tauri Mock` / 桌面 API mock
- 版本与更新检查的 mock manifest
- 文件系统测试夹具与脱敏日志样例

### 5.2 必测主链路

1. 旧版本数据库升级到 `v1.0` 后正常打开主面板
2. 损坏数据库启动时自动恢复并记录日志
3. `macOS` 权限不足时展示引导，授权后恢复可用
4. 关于页展示版本信息并可执行检查更新
5. 长列表场景下主面板仍可选中、粘贴、删除记录
6. 清空历史后数据库与图片资源一致

## 6. 测试数据与夹具规范

### 6.1 数据夹具

建议新增以下夹具类型：

- `seed-release-info.json`：版本与平台信息
- `seed-update-available.json` / `seed-update-latest.json` / `seed-update-error.json`
- `seed-diagnostics-snapshot.json`
- `seed-large-history.json`：包含 `200+` 混合记录的长列表夹具
- `seed-corrupted-db/`：损坏数据库样例

### 6.2 资产夹具

- 使用脱敏文本和脱敏图片
- 图片夹具至少包含横图、竖图、超长图、损坏图各一类
- 不提交真实客户数据、真实路径或真实日志内容

### 6.3 脱敏要求

- 日志样例中的路径、用户名、主机名必须脱敏
- 文本记录夹具不得包含真实密码、Token、邮箱验证码
- 文件路径使用通用示例路径，如 `/Users/demo/Documents/report.pdf`

## 7. CI 建议执行顺序

1. `pnpm run lint`
2. `pnpm run type-check`
3. `pnpm run test:coverage`
4. `cargo fmt --manifest-path src-tauri/Cargo.toml --all -- --check`
5. `cargo clippy --manifest-path src-tauri/Cargo.toml --all-targets --all-features -- -D warnings`
6. `cargo test --manifest-path src-tauri/Cargo.toml`
7. `pnpm run test:e2e`

---

## 8. 通过标准（Definition of Done）

- `P0` 用户故事均有自动化测试覆盖。
- 数据库迁移、恢复、日志、更新检查与清理链路具备单元或集成测试保护。
- 前端覆盖率不低于既有阈值，且新增页面不出现空白测试区。
- 三平台主要回归场景在 mock 或真实环境中至少通过一轮。
- 所有静态检查与自动化命令在主分支可稳定通过。
