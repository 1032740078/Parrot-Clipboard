# v1.0 数据库设计文档

## 1. 文档概述

### 1.1 版本范围

本文档仅描述 `v1.0 Release` 当前版本的存储设计。

`v1.0` 在 `v0.5` 的三类历史记录与分组配置基础上，重点补齐以下能力：

- `SQLite` 版本迁移机制与损坏数据库恢复
- 本地日志目录落盘与诊断读取
- 孤立图片文件扫描与清理
- 长列表与图片场景下的查询性能收口

### 1.2 设计原则

1. **业务数据与诊断数据分离**：历史记录仍存储在 `SQLite`，日志落在独立文件目录。
2. **升级可恢复**：迁移失败或数据库损坏时，优先备份原数据，再尝试重建。
3. **结构稳定优先**：`v1.0` 不为发布质量能力额外引入新的业务表。
4. **资产一致性**：数据库记录、图片文件、清空历史和孤立文件清理边界必须一致。
5. **跨平台可落地**：目录布局与清理策略在 `macOS / Windows / Linux` 上保持一致心智。

## 2. 存储布局总览

### 2.1 运行时目录

| 介质 | 默认位置（逻辑） | 用途 |
|------|------------------|------|
| `clipboard.db` | `app_data/clipboard.db` | 历史记录主库 |
| `config.json` | `app_config/config.json` | 用户偏好配置 |
| `images/original/` | `app_data/images/original/` | 原图文件 |
| `images/thumbs/` | `app_data/images/thumbs/` | 缩略图文件 |
| `logs/` | `app_log/` | 后端、前端、panic 诊断日志 |
| 自启动产物 | 平台系统目录 | 系统集成文件，不落在业务数据库中 |

### 2.2 存储分工

| 数据类型 | 存储位置 | 说明 |
|----------|----------|------|
| 文本 / 图片 / 文件历史 | `SQLite` | 主业务数据 |
| 图片二进制 | 图片目录 | 减少数据库体积，提升清理灵活性 |
| 用户偏好 | `config.json` | 配置读写轻量、可迁移 |
| 错误与诊断日志 | `logs/` | 便于排查与问题反馈 |
| 窗口显隐、当前会话状态 | 内存运行态 | 瞬时状态，不落盘 |

## 3. SQLite 业务模型

### 3.1 版本结论

- `v1.0` 延续 `v0.5` 的三表业务结构：`clipboard_items`、`image_assets`、`file_items`
- `v1.0` 的核心变化不在于新增业务表，而在于 **正式启用迁移框架、损坏恢复与清理机制**
- 当前数据库结构版本通过 `PRAGMA user_version` 表示，**当前版本为 `2`**

### 3.2 主表 `clipboard_items`

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | `INTEGER` | `PK AUTOINCREMENT` | 记录主键 |
| `content_type` | `TEXT` | `NOT NULL` | `text / image / files` |
| `content_hash` | `TEXT` | `NOT NULL` | 去重哈希 |
| `text_content` | `TEXT` | 可空 | 文本正文 |
| `rich_content` | `TEXT` | 可空 | 富文本正文 |
| `preview_text` | `TEXT` | `NOT NULL` | 卡片预览文本 |
| `search_text` | `TEXT` | `NOT NULL` | 为后续搜索预留的统一检索文本 |
| `source_app` | `TEXT` | 可空 | 来源应用名 |
| `file_count` | `INTEGER` | `NOT NULL DEFAULT 0` | 文件类条目数量 |
| `payload_bytes` | `INTEGER` | `NOT NULL DEFAULT 0` | 负载大小估算 |
| `created_at` | `INTEGER` | `NOT NULL` | 首次入库时间 |
| `last_used_at` | `INTEGER` | `NOT NULL` | 最近使用时间 |

**约束**：
- `UNIQUE(content_type, content_hash)`：避免重复内容反复入库
- 列表默认按 `last_used_at DESC, id DESC` 排序

### 3.3 子表与索引

#### 子表 `image_assets`

| 字段 | 类型 | 说明 |
|------|------|------|
| `item_id` | `INTEGER` | 对应 `clipboard_items.id`，主键 |
| `original_path` | `TEXT` | 原图路径 |
| `thumbnail_path` | `TEXT` | 缩略图路径 |
| `mime_type` | `TEXT` | 图片 MIME |
| `pixel_width` / `pixel_height` | `INTEGER` | 图片尺寸 |
| `byte_size` | `INTEGER` | 文件大小 |
| `thumbnail_state` | `TEXT` | `pending / ready / failed` |
| `created_at` | `INTEGER` | 资源创建时间 |

#### 子表 `file_items`

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `INTEGER` | 主键 |
| `item_id` | `INTEGER` | 所属记录 |
| `sort_order` | `INTEGER` | 文件显示顺序 |
| `path` | `TEXT` | 文件或目录路径 |
| `display_name` | `TEXT` | 展示名称 |
| `entry_type` | `TEXT` | `file / directory` |
| `extension` | `TEXT` | 扩展名，可空 |
| `created_at` | `INTEGER` | 创建时间 |

#### 关键索引

| 索引名 | 目标 | 用途 |
|--------|------|------|
| `idx_clipboard_items_last_used_at` | `clipboard_items(last_used_at DESC, id DESC)` | 最近历史列表查询 |
| `idx_clipboard_items_content_type_last_used_at` | `clipboard_items(content_type, last_used_at DESC, id DESC)` | 按类型保留策略与清理 |
| `idx_image_assets_thumbnail_state` | `image_assets(thumbnail_state, item_id)` | 缩略图补偿更新 |
| `idx_file_items_item_id_sort_order` | `file_items(item_id, sort_order)` | 文件详情读取 |

### 3.4 数据库迁移与恢复机制

#### 启动迁移流程

1. 打开 `clipboard.db`
2. 读取 `PRAGMA user_version`
3. 若小于当前 `CURRENT_SCHEMA_VERSION`，按顺序执行 `MIGRATIONS`
4. 每次迁移成功后更新 `user_version`
5. 成功后允许上层继续加载历史记录

#### 损坏恢复流程

当识别到以下错误特征时，触发恢复流程：

- `file is not a database`
- `database disk image is malformed`
- `malformed database schema`
- `unsupported file format`
- `notadb`

恢复策略：

1. 将原始 `clipboard.db` 重命名为带时间戳的 `corrupt` 备份
2. 同步备份 `-wal`、`-shm` sidecar 文件
3. 重建新的 `clipboard.db`
4. 重新初始化表结构与索引
5. 在日志中记录恢复结果，供关于页 / 诊断页展示摘要

### 3.5 为什么 v1.0 不新增业务表

- 权限引导、更新检查、日志、诊断本质上属于 **运行信息与发布质量能力**，不是剪贴板业务历史。
- `Release` 相关信息更适合通过运行态读模型或日志文件提供，而不是写入 `SQLite` 主业务库。
- 保持业务表不变可以让 `v1.0` 升级路径更平滑，降低迁移风险。

## 4. 配置存储设计（`config.json`）

### 4.1 `v1.0` 结构结论

- `v1.0` 继续沿用 `v0.5` 的 `config_version=2`
- 当前版本不新增新的配置分组
- 关于页、日志目录、数据库版本、更新结果等诊断信息 **不写入** `config.json`

### 4.2 配置字段说明

| 分组 | 关键字段 | 用途 |
|------|----------|------|
| `general` | `theme`、`language`、`launch_at_login` | 主题、语言、自启动 |
| `history` | `max_text_records`、`max_image_records`、`max_file_records`、`max_image_storage_mb`、`capture_images`、`capture_files` | 容量与采集策略 |
| `shortcut` | `toggle_panel` | 调出主面板快捷键 |
| `privacy` | `blacklist_rules[]` | 黑名单规则 |

### 4.3 不落盘的运行态

以下状态只保存在运行态内存，不写入 `SQLite` 或 `config.json`：

- 当前主面板是否可见
- 当前是否处于监听暂停态
- 最近一次更新检查是否正在进行
- 当前权限引导弹窗是否打开
- 当前虚拟列表滚动位置与卡片选中状态

## 5. 图片资产与清理边界

### 5.1 图片写入链路

1. 采集图片后先保存原图到 `images/original/`
2. 生成 `200x200` 以内的缩略图到 `images/thumbs/`
3. 将路径与元数据写入 `image_assets`
4. 主面板默认渲染缩略图，原图在粘贴或显式读取时加载

### 5.2 孤立图片清理

清理逻辑通过“目录扫描 + 数据库引用对比”完成：

- 目录存在但 `image_assets` 无引用 → 视为孤立文件，可删除
- `image_assets` 有引用但磁盘文件不存在 → 记为告警，不自动删除数据库记录
- 清理结果需要写日志，返回删除文件数与失败数

### 5.3 清空历史边界

`clear_history` 的清理范围包括：

- `clipboard_items`
- `image_assets`
- `file_items`
- 已关联的图片原图与缩略图文件

不清理以下内容：

- `config.json`
- 日志目录
- 自启动产物
- 平台权限设置

## 6. 性能与可靠性目标

| 目标项 | 指标 |
|--------|------|
| 最近历史查询 | 常用场景下 `< 50ms` |
| 主面板首屏可交互 | `< 200ms` |
| 图片缩略图读取 | `< 50ms` |
| 空闲内存占用 | `< 60MB` |
| 空闲 CPU 占用 | `< 0.5%` |
| 数据恢复 | 异常库可自动备份并重建 |
| 资产一致性 | 清空历史后数据库与图片目录一致 |

---

## 7. 与 v0.5 的主要差异

- `v0.5` 已完成分组配置与三表结构；`v1.0` 不再扩表，而是补齐迁移、恢复、日志与清理机制。
- `v1.0` 将 `PRAGMA user_version`、损坏库备份重建与孤立文件扫描视为正式发布必备能力。
- `v1.0` 更强调“结构稳定 + 异常可恢复 + 资产一致”，而不是继续扩展新的历史数据类型。
