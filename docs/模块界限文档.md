# 粘贴板记录管理工具 - 模块界限文档

## 1. 文档概述

### 1.1 目的

本文档在《模块拆分文档》的基础上，进一步明确各模块的**权责边界**，定义每个模块"拥有什么"、"不负责什么"，以及模块间的**数据所有权**和**接口契约**，防止模块职责蔓延和边界模糊，为团队协作和代码 Review 提供明确的判断依据。

### 1.2 阅读前提

请先阅读以下文档：
- `docs/模块拆分文档.md`：了解各模块的基本职责和接口设计
- `docs/核心功能可行性评审.md`：了解技术选型背景

### 1.3 术语约定

| 术语 | 含义 |
|------|------|
| **拥有（Own）** | 该数据或逻辑只能由本模块创建、修改、删除 |
| **读取（Read）** | 可以查询，但不可修改 |
| **调用（Call）** | 通过接口触发另一个模块的行为 |
| **监听（Listen）** | 被动接收另一个模块发出的事件 |
| **禁止（Forbidden）** | 明确不允许的行为，属于越界 |

## 3. 各模块边界详细说明

### 3.1 M01 - 粘贴板监听模块

#### 边界定义

**拥有：**
- 粘贴板轮询/事件循环的生命周期（启动、停止、暂停、恢复）
- `监听开关状态`（`is_running: bool`）的真实值
- 从系统粘贴板读取原始数据的逻辑

**职责范围内：**
- 检测粘贴板内容是否发生变化
- 将原始粘贴板数据封装为 `ClipboardEntry`
- 过滤黑名单应用的内容（读取 M09 配置）
- 在粘贴板变化时，调用 M02 存储、调用 M07 处理图片

**明确不负责：**
- **禁止**直接操作数据库，存储必须通过 M02 的接口
- **禁止**直接写入图片文件，图片处理必须通过 M07 的接口
- **禁止**直接向前端发送事件，事件推送通过 M13 的接口
- **禁止**读取或修改系统粘贴板用于粘贴（粘贴写入由 M06 负责）
- **禁止**在暂停期间仍然记录内容（M06 调用暂停接口期间应完全静默）

**边界场景举例：**
- 用户执行粘贴操作时，M06 调用 `M01.pause()`，此时 M01 不应产生任何新记录，即使粘贴板内容发生了变化
- M01 检测到图片内容时，只负责提取图片原始字节数据，**不负责**决定缩略图尺寸或存储路径，这些由 M07 决定

---

### 3.2 M02 - 数据持久化模块

#### 边界定义

**拥有：**
- SQLite 数据库文件的读写权
- 数据库 Schema（表结构、索引）的定义与迁移
- 记录的 CRUD 生命周期

**职责范围内：**
- 记录的插入、查询、删除、清空
- 数据库版本迁移（Schema 升级）
- 自动清理过期记录（按数量/大小策略，策略来自 M09）
- 为图片记录存储文件路径引用（路径由 M07 提供）

**明确不负责：**
- **禁止**自行读取或写入图片文件（图片文件操作属于 M07）
- **禁止**自行生成图片路径或缩略图（路径由 M07 决定后传入）
- **禁止**决定清理策略的具体参数，参数必须从 M09 读取
- **禁止**直接触发前端状态更新，删除/新增记录后应通知 M13，由 M13 向前端推送事件
- **禁止**持有超出必要时间的数据库连接（需考虑并发安全）

**数据一致性责任：**
- 当删除 `clipboard_items` 中的记录时，M02 负责通过数据库外键级联删除 `clipboard_images` 和 `clipboard_files` 中的关联行
- 但对应的**图片物理文件**的删除不属于 M02 的职责，M02 应在删除记录前，先通知/返回图片路径，由 M07 执行文件删除

---

### 3.3 M03 - 窗口管理模块

#### 边界定义

**拥有：**
- 主面板窗口的创建参数（大小、位置、透明度、层级）
- 窗口可见状态（`is_visible: bool`）的真实值

**职责范围内：**
- 显示、隐藏、定位主面板窗口
- 多显示器场景下，在活动显示器底部定位
- 窗口几何计算（宽高、边距适配）

**明确不负责：**
- **禁止**决定"何时"显示或隐藏窗口，时机由 M04（快捷键触发）和 M06（粘贴后隐藏）发起
- **禁止**关心窗口内的内容（内容由前端 M05 负责）
- **禁止**直接处理用户的键盘/鼠标事件（事件处理属于前端 M05）
- **禁止**管理设置窗口，设置窗口是独立的 Tauri 窗口，不属于本模块

---

### 3.4 M04 - 快捷键管理模块

#### 边界定义

**拥有：**
- 全局快捷键的注册/注销状态
- `ShortcutAction` 枚举与具体行为的映射关系

**职责范围内：**
- 注册、更新、注销全局快捷键
- 监听系统键盘事件，识别快捷键触发
- 快捷键冲突检测，并返回错误
- 读取 M09 中的快捷键配置，并在配置变更时重新注册

**明确不负责：**
- **禁止**自行执行快捷键触发后的业务逻辑（如显示窗口），必须调用 M03 的接口，保持单向依赖
- **禁止**管理窗口内部（面板内）的键盘操作，面板内快捷键（方向键、Enter 等）属于前端 M05 的事件处理
- **禁止**感知具体平台权限细节，权限检测属于 M14

---

### 3.5 M05 - 主面板 UI 模块

#### 边界定义

**拥有：**
- 面板内所有 UI 组件的渲染逻辑
- 面板内的键盘事件处理（方向键、Enter、Delete、Esc 等）
- 卡片选中状态的视觉呈现

**职责范围内：**
- 渲染记录卡片列表（文本/图片/文件）
- 处理用户的键盘和鼠标交互
- 将用户操作意图（如"粘贴"）转发给 M12 状态管理
- 通过 M12 读取记录数据进行渲染

**明确不负责：**
- **禁止**直接调用 IPC（invoke）获取数据，所有数据必须通过 M12 的 store 读取
- **禁止**自行维护记录列表的副本，M12 是唯一的前端数据源（Single Source of Truth）
- **禁止**决定粘贴的具体实现方式（格式处理、写入系统剪贴板），这些属于后端 M06
- **禁止**管理设置界面的状态，设置界面属于 M10

---

### 3.6 M06 - 粘贴执行模块

#### 边界定义

**拥有：**
- 粘贴操作的完整执行流程（暂停监听 → 写粘贴板 → 隐藏窗口 → 模拟按键 → 恢复监听）
- `PasteMode`（原格式 / 纯文本）的处理逻辑

**职责范围内：**
- 将指定记录内容写入系统粘贴板
- 控制 M01 的暂停/恢复（避免写入粘贴板时自我触发新记录）
- 调用 M03 隐藏窗口
- 通过平台层 M14 模拟 `Ctrl/Cmd+V` 按键

**明确不负责：**
- **禁止**直接查询或操作数据库，记录数据必须由调用方（通过 M13）传入，而不是 M06 自行去 M02 查询
- **禁止**在粘贴成功后删除记录（粘贴不等于消费，记录仍然保留）
- **禁止**关心粘贴的目标应用是哪个（目标应用由操作系统管理）
- **禁止**在模拟按键前额外等待超过必要延迟（延迟应在 M14 平台适配层处理，不硬编码在此模块）

---

### 3.7 M07 - 图片处理模块

#### 边界定义

**拥有：**
- 图片原图和缩略图的文件系统路径命名规则
- 图片文件的物理读写和删除操作

**职责范围内：**
- 将图片字节数据保存为原图文件
- 异步生成缩略图（不阻塞调用方）
- 删除图片文件（原图 + 缩略图）
- 清理孤立图片（数据库记录已删除但文件仍存在的情况）
- 返回保存后的文件路径给调用方（M01 / M02）

**明确不负责：**
- **禁止**自行决定是否应该保存某张图片（超大图片的"是否保存"策略由 M01 结合 M09 配置决定，M07 只负责执行保存）
- **禁止**操作数据库，数据库中的图片路径由 M02 负责维护
- **禁止**直接向前端发送缩略图数据（前端读取缩略图通过文件协议或 Tauri 资产协议）
- **禁止**在生成缩略图时压缩原图

---

### 3.8 M08 - 自启动管理模块

#### 边界定义

**拥有：**
- 操作系统自启动条目的注册和注销

**职责范围内：**
- 启用/禁用开机自启动
- 查询当前自启动状态

**明确不负责：**
- **禁止**自行决定是否启用自启动，这由用户在设置界面操作，由 M09 保存偏好，M08 只负责执行
- **禁止**在每次应用启动时自动检查并修正自启动状态（防止与用户意愿冲突）
- **禁止**关心应用的启动参数或启动后的行为

---

### 3.9 M09 - 配置管理模块

#### 边界定义

**拥有：**
- `config.json` 文件的读写
- `AppConfig` 结构体的完整定义和默认值

**职责范围内：**
- 加载、保存、重置配置
- 为其他模块提供只读的配置查询接口
- 配置变更时，发出变更通知（供 M04、M01 等响应）

**明确不负责：**
- **禁止**校验业务规则（如快捷键是否冲突），业务校验由调用方负责
- **禁止**主动推送配置变更给所有模块，应采用观察者模式：各模块注册感兴趣的配置项变更事件
- **禁止**包含任何 UI 逻辑，配置管理是纯后端模块
- **禁止**在运行时缓存与实际文件不一致的配置（写入即同步）

---

### 3.10 M10 - 设置界面模块

#### 边界定义

**拥有：**
- 设置页面的所有 UI 组件
- 设置表单的本地临时状态（用户编辑中但未保存的状态）

**职责范围内：**
- 展示当前配置值
- 收集用户输入并进行前端校验（如快捷键格式合法性）
- 调用 M12 的 `updateConfig` 方法提交配置变更

**明确不负责：**
- **禁止**直接 invoke 后端命令，必须通过 M12
- **禁止**自行实现快捷键录制逻辑的底层细节（底层由 M04 提供的辅助接口支持）
- **禁止**管理"清空历史记录"操作的实际执行，执行逻辑在后端 M02，调用链为 M10 → M12 → M13 → M02

---

### 3.11 M11 - 系统托盘模块

#### 边界定义

**拥有：**
- 系统托盘图标的渲染状态（正常/暂停）
- 托盘右键菜单的构建和刷新

**职责范围内：**
- 初始化托盘图标和菜单
- 响应托盘菜单点击事件，调用对应模块接口
- 根据监听状态（M01）更新图标样式

**明确不负责：**
- **禁止**直接修改 M01 的监听状态，必须通过 M01 的公开接口（`pause()` / `resume()`）
- **禁止**直接操作窗口内容，显示/隐藏窗口调用 M03 接口
- **禁止**包含数据清空的实现逻辑，"清空历史"菜单点击后，调用 M02 的接口执行

---

### 3.12 M12 - 前端状态管理模块

#### 边界定义

**拥有：**
- 前端所有全局状态（记录列表、选中索引、配置副本、面板可见性等）
- 前端状态的更新规则（action / reducer 等效逻辑）

**职责范围内：**
- 封装所有对 M13 的 IPC 调用，为前端组件提供简洁的异步 action
- 监听 M13 的事件，响应后端推送并更新本地状态
- 维护前端状态的一致性（乐观更新 + 失败回滚）

**明确不负责：**
- **禁止**在 store 中包含任何 UI 组件逻辑（如 DOM 操作、样式判断）
- **禁止**在 store 内部直接调用 `window.__TAURI__`，必须通过 M13 封装的函数
- **禁止**自行持久化前端状态（前端状态是后端数据的投影，应从后端读取恢复，而非缓存在 localStorage）
- **禁止**在前端状态中保存图片的完整二进制数据，只保存文件路径字符串

---

### 3.13 M13 - IPC 通信模块

#### 边界定义

**拥有：**
- Tauri Command（invoke）和 Event（listen/emit）的命名定义
- 前后端数据类型的序列化规范

**职责范围内：**
- 封装所有 `invoke` 调用，提供类型安全的 TypeScript 函数
- 封装所有 `listen` 调用，提供带类型参数的事件订阅函数
- 统一处理 IPC 错误（网络/进程通信错误），与业务错误区分

**明确不负责：**
- **禁止**在 M13 内部包含业务逻辑（如"判断是否需要刷新列表"），业务逻辑在 M12
- **禁止**在 M13 内部维护任何状态
- **禁止**在后端 Rust 侧直接跨越 IPC 层调用另一个模块（后端模块间通信通过直接函数调用或 channel，不应绕道前端）
- **禁止**定义不在接口清单中的临时 Command，所有 Command 必须有明确的业务含义

---

### 3.14 M14 - 平台抽象模块

#### 边界定义

**拥有：**
- 各平台系统 API 的封装实现
- 平台差异的条件编译逻辑

**职责范围内：**
- 提供统一 trait 接口：粘贴板读写、按键模拟、活动应用名获取
- 隔离 `#[cfg(target_os)]` 条件编译，上层模块不感知平台

**明确不负责：**
- **禁止**包含任何业务逻辑（如"粘贴后等待多少毫秒"，时机控制在 M06）
- **禁止**直接调用其他业务模块（M14 是最底层模块，不依赖任何业务模块）
- **禁止**在平台实现中处理错误后静默忽略，必须返回 `Result` 让上层决策

## 4. 模块间交互边界

### 4.1 调用方向约束

```
前端方向（只允许向下调用）：
M05/M10 → M12 → M13 → [后端]

后端方向（层级依赖，禁止逆向）：
M01/M04/M06/M07/M08/M11 → M09（配置读取）
M01/M03/M04/M06/M08     → M14（平台调用）
M01                     → M02（存储）
M01                     → M07（图片处理）
M06                     → M01（暂停/恢复）
M06                     → M03（隐藏窗口）
M11                     → M03（显示/隐藏）
M11                     → M01（暂停/恢复）
```

### 4.2 禁止的调用关系

| 禁止方向 | 原因 |
|----------|------|
| M02 → M07 | 数据层不应依赖处理层，路径由 M07 生成后传入 M02 |
| M14 → 任何业务模块 | M14 是最底层，禁止反向依赖 |
| M09 → 任何业务模块 | 配置是基础服务，禁止反向依赖 |
| M05/M10 → M13（直接调用） | 前端组件必须通过 M12 中转 |
| M13 → M12（后端推送后） | 后端事件由 M12 的监听者处理，M13 只负责传递 |
| M06 → M02（直接查询记录） | 粘贴所需数据由 IPC 调用方传入，M06 不查库 |

## 5. 错误处理边界

### 5.1 错误归属原则

| 错误类型 | 归属模块 | 处理方式 |
|----------|----------|----------|
| 粘贴板读取失败 | M01 | 记录日志，跳过本次变更，不崩溃 |
| 数据库写入失败 | M02 | 返回 `Err`，由 M01 决定是否重试 |
| 图片文件写入失败 | M07 | 返回 `Err`，该条记录不保存 |
| 快捷键注册冲突 | M04 | 返回 `Err`，由设置界面提示用户 |
| IPC 序列化失败 | M13 | 统一包装为 IPC 错误，上报给前端 |
| 前端业务错误 | M12 | Toast 提示，不影响其他操作 |
| 平台 API 不可用 | M14 | 返回 `Err` + 降级标志，上层决策 |

### 5.2 跨模块错误传播规则

- 后端模块统一使用 `Result<T, AppError>` 作为返回类型
- `AppError` 包含错误类型（业务错误 / 系统错误）和来源模块信息
- **禁止**在中间层模块（如 M06）将 `Err` 静默转为 `Ok`，必须向上传播
- 前端 M13 负责将后端 `AppError` 映射为前端可展示的错误消息

## 6. 数据流边界（核心流程）

### 6.1 复制内容被记录的数据流

```
系统粘贴板变化
    │
    ▼
M14::PlatformClipboard::read()         [M14 读取原始数据]
    │
    ▼
M01 解析内容类型，构造 ClipboardEntry  [M01 封装数据]
    │
    ├──（图片类型）──▶ M07::save_original()  → 返回原图路径
    │                  M07::generate_thumbnail() → 返回缩略图路径
    │
    ▼
M02::insert(entry)                     [M02 写入数据库]
    │
    ▼
M13::emit("clipboard:new-record")      [M13 推送前端事件]
    │
    ▼
M12 监听事件，更新 records 列表         [M12 更新状态]
    │
    ▼
M05 响应状态变更，重新渲染卡片列表      [M05 刷新UI]
```

### 6.2 用户粘贴记录的数据流

```
用户在 M05 按下 Enter
    │
    ▼
M05 调用 M12::pasteRecord(id, mode)
    │
    ▼
M12 调用 M13::invoke("paste_record", {id, mode})
    │
    ▼
后端 M06::paste(entry, mode)
    │
    ├── M01::pause()                   [暂停监听]
    ├── M14::PlatformClipboard::write() [写入粘贴板]
    ├── M03::hide()                    [隐藏窗口]
    ├── M14::PlatformKeySimulator::simulate_paste() [模拟Cmd+V]
    └── M01::resume()                  [恢复监听]
```

### 6.3 配置变更的数据流

```
用户在 M10 修改设置并保存
    │
    ▼
M10 调用 M12::updateConfig(partial)
    │
    ▼
M12 调用 M13::invoke("update_config", {config})
    │
    ▼
M09::save(config)                      [M09 持久化配置]
    │
    ▼
M13::emit("config:changed", config)    [M13 推送变更事件]
    │
    ├── M12 监听，更新前端配置副本
    ├── M04 监听，重新注册快捷键（如快捷键变更）
    └── M01 监听，更新黑名单过滤规则（如黑名单变更）
```

## 7. 边界违规检测清单

在代码 Review 时，可使用以下清单检查边界违规：

### 7.1 后端检查项

- [ ] M01 是否有直接写文件（图片）的代码？（应为违规）
- [ ] M02 是否有操作图片文件系统的代码？（应为违规）
- [ ] M06 是否有直接查询数据库的代码？（应为违规）
- [ ] M07 是否有数据库操作代码？（应为违规）
- [ ] M09 是否有调用其他业务模块的代码？（应为违规）
- [ ] M14 是否有 `use` 任何业务模块的代码？（应为违规）
- [ ] 任何模块是否有 `#[cfg(target_os)]` 在 M14 以外的文件中？（应为违规）

### 7.2 前端检查项

- [ ] M05/M10 组件是否有直接 `invoke(...)` 调用？（应通过 M12）
- [ ] M05/M10 组件是否有自己维护的记录列表副本？（应从 M12 store 读取）
- [ ] M12 store 是否包含 DOM 操作或组件引用？（应为违规）
- [ ] M12 store 是否直接调用 `window.__TAURI__`？（应通过 M13 函数）
- [ ] M13 的函数是否包含业务判断逻辑（if/else 分支）？（应为违规）

## 8. 扩展边界说明

### 8.1 未来功能的边界预留

| 未来功能 | 归属模块 | 说明 |
|----------|----------|------|
| 搜索记录 | M02（后端查询） + M05（搜索栏 UI） | 搜索 API 已在 M02 接口中预留 `search()` |
| 收藏记录 | M02（`is_favorite` 字段已存在） + M10（UI 操作） | 数据层已预留，UI 层后续扩展 |
| 加密存储 | M02 内部实现 | 不影响其他模块，M02 内部透明处理 |
| 云同步 | 新增 M15 云同步模块 | 不改变现有模块边界，M15 订阅 M02 的数据变更 |
| 插件系统 | 新增 M16 插件运行时 | 通过 M13 的 IPC 接口与现有模块交互 |

### 8.2 当前阶段刻意排除的边界

- **富文本完整保留**：M06 在纯文本模式下剥离格式，在原格式模式下原样写回粘贴板，不做富文本的进一步处理（如 Markdown 转换），这属于未来版本范畴
- **数据导出/导入**：当前版本 M02 不暴露批量导出接口，该能力留待 v2.0+ 版本
- **跨设备同步**：当前版本所有数据完全本地存储，M09 不包含任何网络/账号配置项

## 9. 总结

本文档通过以下维度明确了 14 个模块的边界：

| 维度 | 说明 |
|------|------|
| **数据所有权** | 每类数据只有一个模块拥有写权，其他模块只读或通过接口操作 |
| **禁止行为清单** | 每个模块明确列出不允许做的事，防止职责蔓延 |
| **调用方向约束** | 单向依赖，禁止逆向调用，防止循环依赖 |
| **错误归属** | 错误在其来源模块明确定义，不被中间层静默消化 |
| **数据流** | 关键流程的完整数据流，清晰展示各模块的参与边界 |
| **Review 清单** | 可操作的检查项，用于代码审查时识别边界违规 |
