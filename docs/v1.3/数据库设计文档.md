# v1.3 数据库设计文档

## 1. 文档概述

### 1.1 版本范围

本文档只描述 `v1.3` 为支持空格聚焦预览、右键上下文菜单与主面板视觉升级所需的数据设计结论。

### 1.2 设计原则

- 优先复用 `v1.2` 已有 SQLite 业务模型与 `config.json` 结构
- 预览层、菜单层、视觉层尽量作为**运行态内存状态**处理，不引入无必要持久化
- 完整内容预览优先通过既有详情查询按需读取，避免列表初始化阶段过度放大读取量

---

## 2. 存储布局总览

### 2.1 版本结论

`v1.3` **不新增 SQLite 业务表、不新增迁移脚本、不升级配置版本号**。

### 2.2 继续沿用的 SQLite 业务模型

`v1.3` 继续沿用现有记录主表与图片资产关联能力，重点复用以下既有数据：

- 记录摘要：用于主列表卡片展示
- 记录详情：用于居中预览层展示完整内容
- 图片缩略图 / 原图路径：用于图片预览放大展示
- 文件列表明细：用于文件卡片完整信息展示

---

## 3. 运行态数据设计

### 3.1 应进入运行态内存、而非数据库的数据

以下状态只在当前前端会话中有效，不应落库：

| 数据 | 说明 |
|------|------|
| `previewOverlayState` | 当前是否打开预览、目标记录、触发来源、加载状态 |
| `contextMenuState` | 当前右键菜单是否打开、归属记录、坐标锚点、动作集 |
| `cardAppearanceState` | 卡片 hover / selected / previewing 等临时视觉状态 |
| `glassSurfaceSpec` | 版本默认视觉令牌，如透明度、模糊、边框、光效 |

### 3.2 为什么不新增业务表

- 预览层与菜单层本质属于当前窗口会话的交互瞬时状态，关闭窗口后没有保留价值
- 半透明与科技风样式是 `v1.3` 默认视觉方案，而不是用户可编辑配置
- 如果把这些状态写入数据库，会把本应短生命周期的 UI 状态错误提升为业务数据

---

## 4. 配置存储设计（`config.json`）

### 4.1 `v1.3` 结构结论

`v1.3` 不新增以下配置项：

- `preview_enabled`
- `context_menu_enabled`
- `panel_blur_radius`
- `panel_opacity`
- `card_style_variant`

以上能力在本版本中均作为默认内建体验提供，不开放自定义开关。

### 4.2 为什么本版不加入“外观偏好设置”

- 当前版本的目标是先收敛一套更统一的默认设计语言，而不是立即引入样式偏好分叉
- 如果同步加入透明度、模糊度、色彩强度等配置，会明显增加测试组合与回归成本
- 未来若确有用户个性化需求，再评估以单独版本扩展 `appearance` 配置分组

---

## 5. 查询与排序约束

### 5.1 聚焦预览的详情读取策略

- 主列表仍通过 `get_records(limit)` 获取摘要数据，不在列表初始化阶段批量加载详情
- 当用户触发空格预览或菜单预览时，再按需调用 `get_record_detail(id)`
- 若预览目标在等待详情期间被删除，应立即关闭预览并回收加载态

### 5.2 右键菜单动作的数据约束

- 菜单动作的目标始终以当前卡片 `record_id` 为准
- `直接粘贴` 与 `纯文本粘贴` 继续复用 `paste_record(id, mode)`
- `删除记录` 继续复用 `delete_record(id)`，成功后由前端列表或事件同步更新 UI

### 5.3 排序与编号约束

- `v1.3` 不改变 `v1.2` 已定义的可视区域快捷编号规则
- 打开预览或右键菜单不应影响记录排序、`selectedIndex` 的真实顺序语义

---

## 6. 性能与可靠性目标

- 打开预览应优先使用已有摘要信息进行首屏骨架或基础渲染，避免明显阻塞
- 详情读取失败时，预览层应进入局部错误态，不影响主面板其余卡片交互
- 右键菜单状态切换必须是纯前端轻量操作，不能依赖后端往返
- 半透明和模糊视觉效果应控制在现有桌面应用可接受范围内，不新增图片或大体积设计资源依赖

---

## 7. 与 v1.2 的主要差异

- `v1.2` 的图片详情读取主要服务于卡片缩略图回退，`v1.3` 则进一步把详情读取扩展到“完整内容预览”场景
- `v1.3` 新增了更多运行态 UI 状态，但这些状态全部停留在前端内存层，不进入 SQLite 或 `config.json`
- 数据结构仍保持稳定，版本演进成本主要集中在前端交互层，而不是存储层
