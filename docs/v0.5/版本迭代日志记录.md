### 2.2 2026-03-07 `v0.5-config-v2-migration`

**变更类型**：`feat(config)`

**主要变更**：
- 重构 `src-tauri/src/config/schema.rs`，建立 `config_version=2` 的 `general / history / shortcut / privacy` 分组配置结构
- 新增旧版扁平 `config.json` 到分组配置的兼容迁移逻辑，并在读取后原地回写标准格式
- 为损坏配置增加默认值回退与自愈重写逻辑，避免配置解析失败阻塞应用启动
- 将配置读取方同步迁移为新的访问方式，保持历史裁剪、自启动初始化与托盘状态读取可用
- 补充配置迁移与持久化写回的 Rust 单元测试

**产出结果**：
- `config.json` 已具备支撑设置中心继续扩展的稳定结构基础
- 老用户升级到 `v0.5` 时可保留原有上限、快捷键与自启动偏好
- 非法配置文件会自动恢复为可用状态，降低启动阶段失败风险

**验证结果**：
- `cargo fmt --manifest-path src-tauri/Cargo.toml --all` ✅
- `cargo test --manifest-path src-tauri/Cargo.toml config:: -- --nocapture` ✅

### 2.3 2026-03-07 `v0.5-settings-domain-foundation`

**变更类型**：`feat(settings)`

**主要变更**：
- 新增 `src-tauri/src/settings/mod.rs`、`model.rs`、`validator.rs`、`blacklist.rs` 与 `tests.rs`
- 建立 `SettingsProfile` 聚合根，集中承载通用设置、历史策略、快捷键与隐私规则的更新入口
- 落地 `SettingsValidationService`，约束语言、历史上限、图片存储上限与快捷键必填校验
- 落地黑名单规则的归一化、去重、新增、更新、删除逻辑，并保留 `created_at / updated_at` 语义
- 导出 `settings` 模块，为后续 IPC 命令、设置页保存和监听链路接入提供稳定领域基座

**产出结果**：
- 设置领域不再直接散落在 `config/` 或 UI 契约中，而是具备独立聚合与校验边界
- 黑名单规则模型已经具备后续对接设置页表单和监听过滤链路的最小闭环
- 后续实现设置相关 IPC 时可直接复用聚合与校验服务，减少命令层重复逻辑

**验证结果**：
- `cargo fmt --manifest-path src-tauri/Cargo.toml --all` ✅
- `cargo test --manifest-path src-tauri/Cargo.toml settings:: -- --nocapture` ✅

## 3. 当前版本结论

### 已锁定事项

- `v0.5` 不提前引入搜索、收藏、云同步、导入导出
- `v0.5` 新增设置窗口、自定义快捷键、黑名单规则、主题与容量策略
- `Linux Wayland` 采用“能力探测 + 明确降级提示”，不做假支持
- `config.json` 升级为分组化、版本化结构，历史记录 SQLite 结构保持不变
- 配置损坏时允许自动回退默认值并原地自愈，优先保证应用可启动
- 设置修改统一经 `SettingsProfile` 聚合与校验服务处理，避免命令层重复实现规则

### 下一步建议

1. 继续完成平台能力矩阵与 `get_platform_capabilities`，为跨平台设置提示建立统一运行时快照
2. 补上设置窗口单实例打开 / 激活能力，尽早打通设置中心的容器层
3. 后续在 IPC 层接入 `SettingsProfile`，复用黑名单和校验逻辑推进保存链路
