# v1.1 数据库设计文档

## 1. 文档概述

### 1.1 版本范围

`v1.1` 仅聚焦主面板窗口行为与键盘交互增强，**不引入新的业务持久化需求**。因此本版本数据库设计的核心结论是：**沿用 `v1.0` 的三表模型与配置结构，不新增业务表、不新增迁移版本、不新增配置字段。**

### 1.2 设计原则

- 运行态问题优先用 **内存状态 + 事件同步** 解决，不把短生命周期状态写入数据库
- `Dock` 覆盖、自动隐藏、自动滚动都属于 **窗口 / UI 行为**，不应下沉到业务持久化层
- 快速粘贴复用现有历史记录主键与排序语义，不引入“快捷键映射表”

---

## 2. 存储布局总览

### 2.1 版本结论

| 存储域 | 结论 | 说明 |
|--------|------|------|
| `SQLite` | 无结构变化 | 继续复用 `clipboard_items / image_assets / file_items` |
| `config.json` | 无结构变化 | 本版不提供“自定义快贴修饰键”等设置项 |
| 运行态内存 | 有变化 | 需要承接面板隐藏原因、当前显隐同步、视口滚动状态 |
| 图片文件目录 | 无变化 | 与 `v1.0` 保持一致 |

### 2.2 继续沿用的 SQLite 业务模型

- 主表 `clipboard_items`：保存记录公共字段与排序依据
- 子表 `image_assets`：保存图片元数据与原图 / 缩略图路径
- 子表 `file_items`：保存文件与目录路径明细

`v1.1` 的 `Command + 数字` 只是对“当前排序后的前 9 条记录”的快捷访问，不需要任何额外持久化字段。

---

## 3. 运行态数据设计

### 3.1 应进入运行态内存、而非数据库的数据

| 数据项 | 建议存储位置 | 原因 |
|--------|--------------|------|
| `panel_visible` | 运行态内存 | 只对当前会话有效 |
| `last_hidden_reason` | 运行态内存 | 调试价值高，但无长期业务价值 |
| `selected_index` | 前端内存 | 主面板关闭后无需恢复 |
| `scroll_left / viewport_width` | 前端局部状态 | 纯渲染行为，不属于业务状态 |
| `previous_frontmost_app_pid` | 后端运行态 | 仅用于恢复焦点 |

### 3.2 为什么不新增业务表

- 自动隐藏不是业务数据，它描述的是“这一次面板为什么收起”
- `Command + 数字` 的目标记录由当前历史顺序直接推导，无需单独映射表
- 自动滚动是视图层策略，不应存成可迁移或可同步的数据
- `Dock` 覆盖是窗口 frame 计算规则，不是存储模型问题

---

## 4. 配置存储设计（`config.json`）

### 4.1 `v1.1` 结构结论

本版本 **不新增配置字段**，继续沿用：

- `general`
- `history`
- `shortcut`
- `privacy`

### 4.2 为什么本版不加入“快贴开关 / 修饰键配置”

- 当前版本目标是恢复高频交互效率，不是扩展设置复杂度
- `Command + 数字` 为明确的单平台优化需求，固定规则更利于验收与回归
- 若后续需要跨平台统一，可在更高版本引入“快贴手势配置”，再评估兼容性与冲突检测

---

## 5. 查询与排序约束

### 5.1 快速粘贴序号来源

- `Command + 1~9` 的目标记录，必须取自主面板当前显示顺序的第 `1~9` 条
- 排序继续沿用既有“最近使用优先”语义
- 粘贴成功后，记录仍通过既有置顶逻辑回到列表首位

### 5.2 不变的查询边界

- `get_records(limit)` 语义保持不变
- `get_record_detail(id)` 语义保持不变
- `paste_record(id, mode)` 语义保持不变，仅新增新的触发入口

---

## 6. 性能与可靠性目标

| 指标 | 目标 |
|------|------|
| 主面板失焦隐藏 | `<= 100ms` |
| `Command + 数字` 触发到粘贴完成 | 常规文本场景 `<= 180ms` |
| 左右切换触发自动滚动 | 体感无明显跳帧，目标 `<= 50ms` |
| 数据库结构变更风险 | `0` 个新业务表 / `0` 条新迁移 |

---

## 7. 与 v1.0 的主要差异

- `v1.0` 重点优化发布稳定性与性能；`v1.1` 重点优化桌面交互，不触碰数据库主模型
- `v1.0` 虚拟滚动已经引入；`v1.1` 只是把“选中项可见性”从实现细节提升为版本约束
- `v1.1` 数据设计的关键不是“新增什么表”，而是明确 **哪些状态绝不能入库**
