- 单个图片文件应直接归入 `image`，单个视频 / 音频 / 文稿文件分别归入 `video / audio / document`。
- 只有在文件集合为混合类型、多目录、多文件组合，或元数据不足以稳定判断时，才回退为普通 `files`。

### 影响

- `v1.4` 的类型识别逻辑从“部分文件类型细分”升级为“单文件统一细分、混合集合统一回退”的稳定模型。
- 图片文件和图片载荷在展示层都能落到 `image` 类型，减少用户理解成本；但它们在底层 `payload_type` 和 `plain_text` 粘贴语义上仍可保持分离。

## 2026-03-10：`v1.4` 的视频 / 音频 / 文稿记录在 `Shift+Enter` 下统一按文件路径列表纯文本粘贴

### 决策

- `v1.4` 的 `plain_text` 粘贴语义继续优先由底层 `payload_type` 决定，而不是由展示层 `semantic_type` 决定。
- 当记录的 `payload_type = files` 时，不论其展示为 `files / video / audio / document`，`Shift+Enter` 与“纯文本粘贴”都统一输出换行分隔的路径列表。

### 影响

- 语义分类和粘贴契约边界更清晰：展示可以更细，但文件类的纯文本粘贴实现仍可复用现有路径输出链路。
- 后续 API、BDD、自动化测试与页面说明都必须统一遵守这条规则，避免出现“看起来是视频卡片，就误以为 `Shift+Enter` 不是路径粘贴”的歧义。

## 2026-03-10：黑底白字截图的 OCR 采用多轮预处理重试，超宽图片卡片优先展示原图

### 背景

- 用户实测反馈，终端路径截图这类黑底白字、小字号图片，即使在预览里看起来很清晰，OCR 仍可能直接返回空结果。
- 这类图片在主面板卡片中还会继续使用小尺寸缩略图并按 `cover` 放大裁切，视觉上更模糊，也会削弱用户对“这张图本来应该能识别”的信心。
- 直接把所有缩略图尺寸做大虽然能改善清晰度，但会显著拖慢后台缩略图生成，不符合当前主链路的性能约束。

### 决策

- OCR 对图片纯文本粘贴改为多轮预处理重试：原图、放大、灰度增强、二值化、反相二值化按顺序尝试，只要任一轮识别出非空文本即返回结果。
- 主面板图片卡片对超宽截图优先解析原图预览，并改用 `object-contain` 展示，避免继续把低分辨率缩略图放大裁切。
- 缩略图生成仍保持当前轻量级尺寸与性能约束，不为了少数高精细截图牺牲整体滚动与异步生成体验。

### 影响

- 黑底白字、小字号路径截图的 OCR 成功率会高于单轮原图识别。
- 图片卡片对“终端截图 / 路径列表截图 / 横向长截图”这类内容会更清晰，但实现方式是前端针对性回退原图，而不是全局放大缩略图。

## 2026-03-10：图片 OCR 改为无损临时 `bmp` 输入，并按识别框位置重建换行

### 背景

- 用户实测发现，图片纯文本粘贴虽然已经接入 OCR，但识别质量仍不稳定，尤其对文字较密、分辨率边缘的图片更敏感。
- 原实现直接把 OCR sidecar 的 `text` 模式输出作为最终结果，这会把引擎内部切分出的文本块直接按换行拼接，导致同一行内容经常被拆成多行。
- 当前跨平台剪贴板层只能拿到原始 RGBA 像素，无法恢复用户最初复制时到底是 `png`、`jpeg` 还是其他编码格式；如果继续纠结“保留原文件格式”，实现上并不成立。

### 决策

- OCR 执行前不再直接把缓存原图路径交给 sidecar，而是基于内存中的原始 RGBA 像素临时写出无损 `bmp` 文件，作为 OCR 输入，避免 OCR 前再叠加一次不必要的编码干扰。
- OCR 输出统一改走 `json` 模式，读取每个文本块的位置信息，再按行坐标重建最终纯文本。
- 最终文本只在识别框明显落入不同行时才插入换行；同一行的文本块默认合并，英文 / 数字块在存在明显水平间距时补单个空格。

### 影响

- “图片看起来被压了一遍后再识别”的体感会下降，OCR 输入更接近剪贴板中的原始像素数据。
- 纯文本粘贴的结果会更贴近图片里的自然排版，不再把同一行碎成很多换行。
- 由于系统剪贴板并不提供可恢复的源编码格式，产品语义应统一表述为“按原始像素做无损 OCR 输入”，而不是“保留用户最初复制时的图片文件格式”。

## 2026-03-09：图片记录的 `Shift+Enter` 纯文本粘贴升级为“面板内 OCR 后再粘贴”，OCR 引擎随包内嵌

### 背景

- 图片 OCR 的执行耗时明显高于普通文本 / 文件纯文本粘贴，如果仍沿用“按下快捷键立刻隐藏面板”的旧语义，用户会在等待期间失去反馈，不知道系统是否仍在工作。

- `Shift+Enter` 命中图片记录时，纯文本粘贴能力正式扩展到图片：后端先执行 OCR，成功后再写入识别文本、隐藏面板并模拟粘贴。

### 影响

- 图片记录现在拥有稳定的“提取文字并直接粘贴”出口，和文本 / 文件记录一起归入 `plain_text` 语义集合。
- 打包产物会额外包含按目标平台选择的 OCR sidecar，但运行态依然保持单包分发，不需要单独安装命令行工具。
- 旧的“图片纯文本粘贴直接禁用”规则被废弃，前端菜单、快捷键提示和后端校验必须统一更新为“图片允许发起 OCR 纯文本粘贴”。

### 背景

- macOS 下窗口移动与缩放直接对原生 `NSWindow` 调用 `setFrame:display:animate:`，不再走 Tauri 的物理坐标 `set_position / set_size` 路径。

### 影响

- 主屏 / 副屏 / 混合缩放来回切换时，不再依赖“当前窗口缩放因子”和“目标显示器缩放因子”的组合正确性，位置错乱概率显著降低。
- 调试日志会同时输出目标 frame 与实际 frame，便于后续继续追查极端桌面布局问题。
- 非 macOS 平台仍保持现有通用定位流程，不扩大平台差异面。

## 2026-03-07：macOS 多显示器定位改为直接走 CoreGraphics 选屏，不再依赖 Tauri 光标坐标

### 背景

- 手动测试发现：主面板在多显示器场景下仍然稳定落在主显示器，说明当前“根据鼠标所在屏幕选中目标显示器”的实现未真正生效。
- 现有实现依赖 `tauri::AppHandle::cursor_position()` 再和 `available_monitors()` 的 `work_area` 做命中判断，但 macOS 下该光标坐标链路来自 `tao`，会混入主屏缩放因子，导致混合缩放显示器场景下坐标失真。

### 决策

- macOS 下主面板选屏改为直接调用 CoreGraphics `CGGetDisplaysWithPoint`，以系统级显示器 ID 判断鼠标当前所在显示器。
- 目标显示器的可用工作区不再依赖 Tauri 反推，而是基于 `CGDisplayBounds + NSScreen.visibleFrame + backingScaleFactor` 直接计算。
- 非 macOS 平台仍保留原有 `cursor_position + available_monitors` 通用逻辑，避免扩大改动面。

### 影响

- 混合缩放或副屏场景下，主面板更有机会准确跟随当前鼠标所在显示器，而不是错误回退到主显示器。
- 定位实现会多一层 macOS 平台分支，但根因被收敛在窗口模块内部，不影响 IPC 与前端契约。
- 后续若 Tauri / tao 修复光标坐标问题，也可以在当前隔离层内切换回官方实现。

# 项目决策记录文档

## 2026-03-06：v0.2 持久化采用 SQLite 三表 + 图片资源文件落盘方案

### 背景

- `v0.2` 需要在 `v0.1` 文本内存历史的基础上新增图片、文件/目录记录，并支持应用重启后恢复历史。
- 图片原始二进制若直接写入 SQLite，会放大数据库体积、增加备份与读取压力，也不利于缩略图的独立清理。
- 文件记录本质上是“一个历史记录包含多个路径项”，需要与主记录建立 1:N 关系。

### 决策

- 历史记录统一采用 `clipboard_items` 主表保存公共字段。
- 图片类型使用 `image_assets` 表保存元数据，原图与缩略图文件存放到应用数据目录。
- 文件类型使用 `file_items` 表保存路径明细，通过外键与主表关联。
- 数据库迁移版本使用 `PRAGMA user_version` 管理，不新增业务外的版本表。

### 影响

- 主面板列表查询可通过主表 + 子表 Join 输出轻量摘要，便于后续扩展。
- 图片记录删除时需要执行“数据库删除 + 文件清理”的组合流程，但可通过异步补偿保证最终一致性。
- 为后续 `v0.3+` 的搜索、筛选、分页提供可扩展数据底座。

## 2026-03-06：v0.2 主界面继续保持单页模型，摘要与详情分离

### 背景

- 当前版本目标聚焦在“复制后快速找回并再次粘贴”，而不是扩展更多管理界面。
- 图片原图路径、文件明细列表等详情数据体积明显大于主面板卡片渲染所需信息，若一次性全部返回会拖慢面板打开速度。

### 决策

- `v0.2` 继续仅保留一个核心页面：主面板，不引入设置页、托盘页、搜索页。
- `get_records` 只返回卡片摘要 DTO，满足主面板首屏渲染。
- 新增 `get_record_detail` 获取完整详情，供粘贴链路和后续扩展按需读取。
- 记录更新采用事件推送，缩略图就绪、复用置顶等场景通过 `clipboard:record-updated` 通知前端增量刷新。

### 影响

- 主面板打开时延可控，图片和文件详情不会拖慢首屏。
- 前端状态模型从“单一文本记录列表”升级为“摘要列表 + 按需详情读取”。
- 后续若引入搜索和详情面板，可复用现有摘要/详情分离契约。

## 2026-03-06：v0.2 默认保留上限设为文本 200、图片 50、文件 100，纯文本粘贴仅限文本

### 背景

- 持久化引入后，历史记录不再随着应用退出自动清空，需要明确容量边界，避免数据库与图片资源无限增长。
- 文本、图片、文件三类内容的体积差异显著，不能复用同一个固定上限。
- `Shift + Enter` 的纯文本模式仅对文本语义成立，对图片和文件没有合理的降级结果。

### 决策

- 默认保留上限设置为：文本 `200`、图片 `50`、文件 `100`。
- 清理策略按内容类型分别执行，仅淘汰对应类型的最旧记录。
- `plain_text` 粘贴模式仅允许 `text` 类型记录使用；前端默认禁用非文本场景，后端使用 `INVALID_PARAM` 兜底。

### 影响

- 持久化空间占用更可控，图片记录数量被重点限制。
- 用户在主面板中的历史结构更稳定，不会因某一类型高频复制挤占所有其他类型。
- API、BDD 与状态码契约可围绕统一的模式限制达成一致。

## 2026-03-06：macOS 粘贴模拟增加焦点稳定等待与按键驻留时长

### 背景

- 在 macOS 自动化验收中发现：历史记录通过 `Enter` 触发粘贴后，系统剪贴板内容已正确更新，但焦点刚从面板切回目标应用时，`Cmd+V` 模拟偶发丢失。
- 现有实现中，面板隐藏后仅等待 `80ms` 就立即发送 `Cmd+V`，并且按键按下与抬起之间没有任何驻留时间，对焦点切换较慢的应用不够稳健。

### 决策

- 将面板隐藏后的“焦点稳定等待”延长到 `140ms`，给目标应用更多时间完成前台激活。
- macOS 按键模拟补充 `20ms` 的按下驻留时长，再发送抬起事件，降低目标应用漏收组合键的概率。

### 影响

- 历史记录粘贴在 TextEdit、Terminal 等需要前台焦点的应用中稳定性更高。
- 粘贴总耗时会略微增加，但仍保持在 v0.1 的交互预算范围内；后续可在 v0.2 E2E 中继续精细量化。

## 2026-03-06：主面板定位改为基于可用工作区，并提升 macOS 原生窗口层级

### 背景

- 手动测试发现：主面板虽然已经开启 `alwaysOnTop`，但在 macOS 下仍会被底部 Dock 遮挡。
- 现有窗口定位逻辑直接使用整块屏幕高度贴底，未扣除 Dock 和菜单栏占用的 `work_area`。
- Tauri 的 `alwaysOnTop` 在 macOS 对应 `NSFloatingWindowLevel`，层级低于 Dock 的 `NSDockWindowLevel`，因此仅靠现有配置不足以保证面板始终浮在系统 UI 之上。

### 决策

- 主面板的宽度、横向起点和纵向位置统一基于显示器 `work_area` 计算，确保不会落入 Dock 或菜单栏占用区域。
- macOS 下在显示主面板时，额外将原生窗口层级提升到 `NSStatusWindowLevel`，高于 Dock。
- 主面板在 macOS 下加入“所有空间可见”与“全屏辅助窗口”行为，保证从不同 Space 或全屏应用唤起时仍能显示在最前。

### 影响

- 面板显示时会紧贴可用桌面区域底部，而不是紧贴物理屏幕最底部。
- Dock 位于底部、左侧或右侧时，面板都会自动避让，不再被系统栏遮挡。
- 从普通桌面、Mission Control 多空间或全屏应用中唤起面板时，显示一致性更高。

## 2026-03-06：重复内容统一复用原记录，粘贴成功后显式置顶

### 背景

- 手动验收发现：当用户从历史卡片中执行粘贴时，监听模块会把这次“程序写入系统粘贴板”的内容再次捕获，导致列表新增一条重复记录。
- 现有 v0.1 设计仅对“最新一条记录”做去重，旧记录再次进入系统粘贴板时仍会生成新 ID，和“历史列表保持简洁”的目标不一致。

### 决策

- 文本去重从“仅比较最新记录”调整为“对整条历史列表做精确匹配”。
- 如果新文本与非最新记录相同，不再新建记录，而是复用原记录并移动到最左侧。
- 用户执行粘贴时，不再依赖监听器误捕获来触发置顶；改为在粘贴成功后由后端显式将对应记录置顶。
- 粘贴流程在写入系统粘贴板后，先同步监听器快照，再恢复监听，避免把应用自身写入再次识别成新变化。

### 影响

- 粘贴历史记录后，列表总数保持不变，但被粘贴的记录会移动到最左侧。
- 用户再次复制一段历史中已存在但当前不是最新的文本时，也会复用原记录并置顶，不再出现重复卡片。
- `paste_record` IPC 返回被置顶后的 `ClipboardRecord`，前端可立即同步本地状态。

## 2026-03-06：`config.json` 采用扁平字段结构，并在启动时自动补齐缺失上限

### 背景

- `v0.2` 需要在原有文本上限之外，再新增图片与文件两类记录上限。
- `v0.1` 的配置模型只有 `max_text_records` 与 `toggle_shortcut`，如果直接切换到严格新结构，旧用户升级后会面临缺字段启动失败或需要手工重建配置的问题。
- 当前版本尚未引入设置页，用户对容量上限的观察与调整仍主要依赖运行时目录下的 `config.json`。

### 决策

- `config.json` 继续采用扁平字段结构：`max_text_records`、`max_image_records`、`max_file_records`、`toggle_shortcut`。
- 应用启动时从运行时目录加载 `config.json`；若文件不存在，则自动创建一份默认配置。
- 读取旧配置时，对缺失的新字段自动回填默认值，并将补齐后的完整配置回写到文件，保证用户能直接看到三类上限配置。

### 影响

- `v0.1` 升级到 `v0.2` 不需要手工删除或重建配置文件，升级路径更平滑。
- 用户可以直接在同一份配置文件中看到文本、图片、文件三类保留上限。
- 后续若补充设置页或配置编辑能力，可以沿用这套配置字段与兼容迁移策略。

- 数据层与系统集成层职责更清晰，后续排查问题时可以快速定位到 SQLite、配置文件或系统文件三类介质。

## 2026-03-07：v0.5 将 `config.json` 从扁平结构升级为分组化 `config_version=2`，黑名单规则继续落在配置层

### 背景

- `v0.3` 之前的 `config.json` 仅包含少量扁平字段，适合没有设置页的阶段；进入 `v0.5` 后，配置项将扩展到主题、容量、快捷键、黑名单规则等多个分组，继续堆叠扁平字段会让语义和迁移成本迅速上升。
- 黑名单规则本质上是用户偏好，而不是业务历史；若为了设置页引入新的 SQLite 业务表，会把历史数据和配置规则耦合在一起，增加迁移与排查成本。
- 现有 SQLite 三表结构已经足以支撑文本、图片、文件历史记录，不需要为了 `v0.5` 的设置能力再扩表。

### 决策

- `config.json` 在 `v0.5` 升级为分组化结构：`general`、`history`、`shortcut`、`privacy`，并增加 `config_version=2`。
- `v0.3` 及更早版本的扁平配置在启动时自动迁移到新结构；旧值保留，新增字段回填默认值，并将迁移结果回写到同一份 `config.json`。
- 黑名单规则继续作为配置数据，落在 `privacy.blacklist_rules` 中，不新增 SQLite 业务表。
- `clipboard.db` 继续只承载历史记录数据；主题、快捷键、黑名单、自启动等设置项统一落在配置层或系统集成产物中。

### 影响

- 设置窗口可以直接按领域分组读写配置，前后端契约更稳定，也更接近用户心智模型。
- 升级到 `v0.5` 时不需要重建数据库，只需执行一次轻量配置迁移，升级路径更平滑。
- 后续如果增加导入导出或设置备份，配置文件可以整体搬运，不需要跨 SQLite 和 JSON 做双向拼装。

## 2026-03-07：v0.5 配置迁移失败时自动回退默认配置，并按平台生成默认面板快捷键

### 背景

- `v0.5` 已将 `config.json` 升级为 `config_version=2` 的分组结构，启动阶段需要同时兼容旧版扁平配置、局部缺失配置以及用户手工修改后的异常配置。

- 若配置解析失败直接中断启动，会让设置中心成为主流程可用性的单点风险。

### 决策

- 读取 `config.json` 失败时，不中断应用启动；直接回退到默认 `v2` 配置，并将自愈后的配置原地回写到同一路径。
- 默认调出面板快捷键按平台区分：`macOS` 使用 `Shift+Command+V`，`Windows` / `Linux` 使用 `Shift+Control+V`。
- 配置文件标准化后的落盘格式统一为 `config_version=2`，不再继续输出旧版扁平字段。

### 影响

### 影响

- 后续 Windows / Linux 真正接入平台 API 时，可以在不改动上层契约的前提下逐步替换底层实现。

- 隐私能力已经从“可配置”升级为“监听阶段真实生效”，敏感应用中的复制内容不会再进入历史数据库。

### 影响

- 主面板首次打开时可以承载更多历史记录，同时不会因一次性渲染过多卡片导致明显卡顿。
- 键盘交互、权限引导与删除逻辑继续复用既有状态模型，降低虚拟滚动引入行为回归的风险。

## 2026-03-08：图片原图缓存采用共享 `LRU` 双阈值策略，最近历史摘要在 SQL 层直接计算文本元数据

### 背景

- `TASK-03-1` 已把主面板历史记录读取上限提升到 `200`，接下来瓶颈从“渲染数量”转向“图片恢复链路和摘要查询负载”。
- 当前原图粘贴每次都会重新打开磁盘文件并解码 `PNG`，高频粘贴同一批截图时会重复消耗 IO 与 CPU。
- 最近历史摘要查询此前会把整段 `text_content` 带回 Rust 再计算字数与行数，长文本场景会产生不必要的数据搬运。

### 决策

- `ImageStorageService` 为原图读取链路新增共享内存 `LRU` 缓存，缓存上限固定为“最多 `32` 张原图 + 最多 `64MB` 解码后字节”，任一阈值超出即按最近最少使用策略淘汰。
- 原图缓存只在真正读取原图时填充；缩略图浏览与图片落盘阶段不让原图常驻内存，保持“按需加载”的发布策略。
- 删除图片资产时同步驱逐对应缓存项，确保磁盘与内存生命周期一致，不给后续孤立文件清理留下隐患。
- 最近历史摘要查询改为由 `SQLite` 直接计算 `text_char_count / text_line_count`，Rust 侧只接收元数据，不再搬运完整正文。

### 影响

- 高频图片粘贴与预览场景会减少重复解码，对长时间运行时的 CPU / IO 抖动更友好。
- 原图缓存具备确定的内存预算，不会因为重复访问截图历史而无限增长。
- 主面板大列表查询在文本记录很多时会更轻，后续 `TASK-03-3` 和 `TASK-04-1` 可以继续围绕真实负载补齐治理与回归。

### 背景

## 2026-03-08：v1.0 发布链路采用“日常 CI 无签名冒烟 + tag 触发正式签名发布”，macOS `DMG` 改为无界面脚本封装

### 背景

- `TASK-04-2` 需要同时收口三平台打包、签名、发布清单与回滚说明，但当前仓库只有单平台 `pnpm tauri build`，缺少正式发布工作流与发布前校验入口。
- Tauri 默认 `DMG` 生成依赖 `create-dmg` 的 Finder AppleScript，美化安装盘界面时在终端 / CI 环境中容易阻塞，不适合作为稳定的自动化发布基线。
- 日常分支验证与正式发布对签名要求不同：CI 需要尽快发现跨平台构建回归，而正式发布才需要导入证书、执行签名与上传 Release 产物。

- 日常 `CI` 采用三平台 `--no-sign` 打包冒烟，只验证能否生成当前平台 bundle；正式发布通过 tag 触发独立 `release.yml` 工作流，执行签名、产物收集和 Draft Release 上传。

- 发布前统一执行 `release:verify-config`、`release:check` 与 `release:build:local`，把版本校验、全量自动化回归与本机打包冒烟收敛为标准发布门禁。

### 影响

- `v1.0` 发布链路具备更清晰的职责分层：日常 CI 更快，正式发布更可控。

### 背景

- `v1.5` 从音效与预览联动进入媒体 / 文稿 / 链接预览阶段后，原有 `ClipboardRecordDetail` 只能稳定承载 `text / image / files` 三类明细，已经不足以表达多类型预览状态与资源引用。
- 当前版本已经明确不扩展底层 `payload_type`，仍保持 `text / image / files` 三类粘贴语义，因此音频、视频、文稿和链接预览必须通过独立的“预览状态 + 资源索引”层补齐，而不是继续把所有字段硬塞回主记录表。
- 后续 `TASK-15-03-2` 到 `TASK-15-04-3` 还需要接入音频 / 视频播放器、PDF / Office 结构化内容和链接摘要，如果现在不先固定底层契约，后续实现会持续返工。

### 决策

- `v1.5` 的预览详情底座正式采用 SQLite schema v4：在 `clipboard_items` 中补齐 `primary_uri`、`preview_renderer`、`preview_status`、`preview_error_code`、`preview_error_message`、`preview_updated_at`。
- 新增 `preview_assets` 通用资源表，统一承接 `document_outline`、`document_text`、`link_summary` 等派生预览资源索引；大体积内容继续走缓存文件路径，小体积结构化内容允许直接以文本形式落索引表。
- 前后端共享契约统一扩展 `preview_status`、`preview_renderer`、`audio_detail`、`video_detail`、`document_detail`、`link_detail`，并约定默认渲染器回退策略：`text -> text`、`image -> image`、`files -> file_list`、`document(pdf) -> pdf`、`document(others) -> document`、`link -> link`。

### 影响

- 后续媒体 / 文稿 / 链接预览实现可以围绕统一的预览状态机推进，而不需要为每种类型各自发散一套独立详情命令和表结构。
- 数据库、Rust 映射层与前端类型系统已经具备继续扩展异步资源准备、失败回传和缓存刷新能力，后续任务将更多聚焦在解析器和渲染器本身。
- 当前这次决策只完成“底座收敛”，并不代表所有多媒体与文稿预览已经可直接使用；相关 UI 渲染、命令和自动化测试仍需在后续任务中补齐。

## 2026-03-14：v1.5 文稿预览放弃 `LibreOffice` 依赖，统一采用应用内开源库与轻量降级策略

### 背景

- 在讨论 `v1.5` 文稿预览方案时，曾评估过通过 `LibreOffice headless` 做 `DOC / DOCX / XLS / XLSX / PPT / PPTX` 转换后再展示的路径。
- 该方案虽然能减少部分格式解析工作，但会引入额外系统依赖、安装环境差异、冷启动开销和缓存清理复杂度，不符合当前项目“轻量桌面工具”的定位。
- 用户已经明确要求当前版本放弃 `LibreOffice` 方案，因此需要把文稿预览策略正式收敛到应用内方案。

### 决策

- `v1.5` 不引入 `LibreOffice` 或其他系统级 Office 转换依赖作为文稿预览前提。
- `PDF` 优先采用专用 PDF 渲染方案；`DOCX / XLSX / PPTX` 优先采用应用内开源库或结构化解析方案；`DOC / XLS / PPT` 只做轻量抽取尝试，失败时明确降级。
- 文稿预览当前版本以“显示主要内容、结构或摘要”为目标，不承诺像素级排版还原。

### 影响

- 打包、安装与运行环境更轻，减少因为外部依赖缺失导致的预览不可用问题。
- 开发重点会转向“预览资源准备、HTML 清洗、结构化内容呈现与降级体验”，而不是维护外部转换进程。
- 旧二进制 Office 格式的成功预览率会低于引入系统转换依赖的方案，但版本边界更清晰，用户体验也更可控。

## 2026-03-14：v1.5 媒体预览统一采用开源播放器层，短音效统一采用专用音频库

### 背景

- 用户进一步确认，视频和音频的预览播放方案也希望优先采用开源库，以减少自研播放器状态管理与兼容细节维护成本。
- 当前项目虽然可以直接基于原生 `audio / video` 标签做最小播放能力，但如果在多个组件里分散维护播放状态、控件 UI、错误回退和后续扩展，会让媒体预览代码快速膨胀。
- 同时，复制 / 粘贴 / 预览成功这类短音效与内容级音频预览是两类不同问题，若混用同一套实现，职责会变得模糊。

### 决策

- `v1.5` 的音频 / 视频内容预览统一采用开源播放器层，当前优先选型为 `Vidstack`，用于承接 `React` 集成、音视频统一播放器外壳与后续扩展能力。
- `v1.5` 的短音效播放统一采用 `Howler.js`，只负责复制成功、粘贴成功、预览首次显示成功等短音效，不承担内容级媒体预览职责。
- `hls.js` 与 `wavesurfer.js` 仅作为后续候选增强项：前者用于未来流媒体场景，后者用于未来音频波形场景，均不进入当前版本基线。

### 影响

- 媒体预览的 UI 与状态管理会更一致，减少在业务组件中重复维护播放器交互逻辑。
- 音效播放与内容预览边界更清晰，后续调试、回归与替换都更容易定位问题。
- 需要明确一点：开源播放器库可以提升播放器层一致性，但不能替代底层 WebView 的编解码能力，因此 `v1.5` 仍需保留“不支持当前编码预览”的降级策略。

## 2026-03-14：v1.5 短音效先落在前端统一音效服务，Rust 继续只负责成功事件源

### 背景

- 在正式实现 `TASK-15-01-2` 时复查仓库，确认当前前端和 Rust 侧都没有现成的原生音频播放基础设施。
- 复制成功、粘贴成功、预览窗口首次可展示这 3 个触发点，当前都已经有稳定的前端订阅点或 IPC 成功返回点可复用。
- `v1.5` 本轮目标是快速补齐明确反馈，同时保证失败静默降级、不阻断主流程，因此优先选择实现成本更低、现有链路更稳定的落地方式。

### 决策

- `v1.5` 的短音效播放统一由前端 `SoundEffectService` 承担，基于 `Howler.js` 懒加载 `copy-notification.mp3`、`paste-notification.mp3`、`preview-open.mp3` 3 个项目内英文资源。
- Rust 保持现有职责边界：复制成功继续通过 `clipboard:new-record` 事件暴露，粘贴成功继续通过 `paste_record` 命令返回暴露，预览首次显示成功继续由预览窗口详情加载状态驱动。
- 音效触发点固定为 `useClipboardEvents` 的新记录事件、`recordPaste` 的成功粘贴分支、`PreviewWindow` 首次进入 `ready` 状态；音效失败只记日志，不改变业务成功与失败判断。

### 影响

- 不需要在当前版本额外引入 Rust 原生音频依赖、跨平台设备兼容与资源桥接逻辑，能更快完成 `v1.5` 的关键反馈目标。
- 业务成功边界仍然由后端领域事件和 IPC 结果决定，不会把复制、粘贴、预览成功语义散落到组件细节里。
- 后续如果要新增静音、音量或自定义音效，也可以继续在统一前端音效服务上扩展。

## 2026-03-14：预览联动采用“显式打开会话 + 轻量记录同步 IPC”，不复用会抢焦点的打开命令

### 背景

- 现有 `show_preview_window` 在窗口已存在时仍会执行 `show + bring_to_front + focus`，直接复用它做联动切换会导致主面板连续浏览时被预览窗口反复抢焦点。
- 当前前端已经有 `previewOverlay` 运行态、主面板 `selectedIndex` 选中态以及预览窗口内部的 `system:preview-window-requested` 事件订阅链路，缺少的只是“窗口已打开时静默切换目标记录”的轻量能力。
- `v1.5` 已经明确“显式打开后联动、关闭后停止联动”，因此需要把“打开会话”和“跟随切换”拆成两个动作，而不是继续复用同一个窗口打开命令。

### 决策

- 预览会话状态以 `previewOverlay` 为真源，并补齐 `followSelection` 语义：只有用户显式打开预览窗口后，该会话才允许跟随主面板选中项变化。
- 新增轻量 IPC `sync_preview_window_record`，仅在预览窗口已存在时向预览窗口发送新的记录 ID，不执行 `show`、`focus` 或提层动作。
- 主面板负责在预览会话存在且 `followSelection = true` 时，把新的选中记录同步给预览窗口；关闭预览窗口后，联动会话随显隐事件一起结束。

### 影响

- 键盘左右切换、数字快选等连续浏览操作可以在不丢失主面板焦点的前提下驱动预览窗口跟随切换，更符合 `v1.5` “连续浏览工作区”的体验目标。
- 显式打开与静默联动的职责边界被清晰拆分，后续如果要加入节流、失败兜底或联动开关，都可以围绕 `followSelection + sync_preview_window_record` 扩展。
- 预览首次打开音效与联动切换逻辑也因此更容易区分：首次成功显示播放音效，联动切换只更新内容、不重复播放打开音效。
