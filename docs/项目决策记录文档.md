## 2026-03-09：`v1.3` 右键菜单状态以 `recordId + anchor + actions` 为唯一真源，并预先固定动作顺序

### 背景

- 右键菜单的核心风险不是“能不能弹出”，而是“弹出来后到底作用于哪一张卡片”。
- 如果菜单目标、定位点、动作清单分散在多个局部状态中，连续右键不同卡片时很容易出现菜单还挂在旧记录上、动作禁用态不同步的问题。

### 决策

- 在 `src/stores/useUIStore.ts` 中新增 `contextMenu` 运行态，统一收敛 `recordId / x / y / placement / collisionAdjusted / actions / openedAt`。
- 菜单动作顺序固定为 `预览完整内容 -> 直接粘贴 -> 纯文本粘贴 -> 删除记录`。
- `纯文本粘贴` 的禁用规则在动作模型层就确定：只有文本卡片可用，图片与文件卡片统一禁用。
- 菜单关闭原因单独记录为 `lastContextMenuCloseReason`，至少覆盖 `click_outside / action_completed / escape / record_deleted / panel_hidden`。

### 影响

- 菜单目标、定位和动作禁用态都有了同一份前端真源，后续右键交互和菜单组件可以围绕这份状态模型实现。
- 动作顺序和禁用规则在 UI 之前先被测试锁定，能减少不同卡片类型下的行为分叉。
- 该决策不改变 IPC，仅影响前端运行态和菜单动作建模。

## 2026-03-09：`v1.3` 预览详情采用按需读取 + 内存缓存 + 局部错误态，不弹全局失败提示

### 背景

- `v1.3` 预览层要展示完整文本、原图和文件明细，但主面板列表仍然只需要轻量摘要；如果在面板初始化时把全部详情一次性拉完，会拖慢打开速度。
- 预览详情失败时，用户仍然应该留在当前主面板场景中，不应因为单条记录详情读取失败就收到全局 Toast 或被迫退出整个操作流。

### 决策

- 预览详情统一在 `src/hooks/usePreviewOverlay.ts` 中按 `recordId` 打开时调用 `get_record_detail` 按需读取。
- 已成功读取过的详情缓存在前端内存 `Map` 中，重复预览同一条记录时优先命中缓存。
- 详情读取失败只把 `previewOverlay` 运行态切到 `error`，由 `PreviewOverlay` 组件显示局部错误卡片，不触发全局 `Toast`。
- 当预览目标记录已经从 `ClipboardStore` 中消失时，预览层自动按 `record_deleted` 语义关闭。

### 影响

- 主面板保持轻量，只有用户真正进入预览时才支付详情查询成本。
- 预览失败的影响范围被限制在当前覆盖层，不会干扰主面板浏览、选中和后续操作。
- 该决策不增加新的 IPC 命令，只在前端补充查询时机、缓存策略与错误反馈方式。

## 2026-03-09：`v1.3` 预览打开时，键盘层优先消费 `Space / Esc` 并抑制主列表快捷键

### 背景

- `v1.3` 预览层属于“先确认再执行”的交互，若预览打开后仍然继续响应 `Enter`、`Delete` 或数字快选，用户会在看内容时误触粘贴、删除或切换记录。
- 预览尚未拆成独立窗口，而是覆盖在主面板之上，因此 `Esc` 的第一职责应该是退出当前预览，而不是直接把整个主面板收起。

### 决策

### 影响

- 键盘交互优先级更加清晰：预览会话先于主列表会话，减少误操作。
- `useKeyboard` 会成为预览键盘闭环的统一入口，后续预览组件只需消费 store 状态，不需要自己重复监听全局按键。
- 该决策不改变 IPC 契约，只影响前端快捷键分发顺序与 UI 运行态。

## 2026-03-09：图片本地资源预览除了 `convertFileSrc` 外，还必须启用 `Tauri assetProtocol` 并收敛到应用图片目录 scope

### 背景

- 真实环境中截图记录已经完成 `ThumbnailReady`，前端也会继续请求 `get_record_detail`，但图片卡片仍退化为“预览不可用”，说明问题不在缩略图生成，而在 WebView 对本地资源 URL 的访问阶段。
- 仅在前端调用 `convertFileSrc` 并不足以保证资源可见；若 `Tauri` 侧未开启 `assetProtocol` 或未把图片目录加入 `scope`，`asset://localhost/...` URL 仍会被拦截，最终触发缩略图与原图双双加载失败。

### 决策

- `Tauri` 配置中显式开启 `app.security.assetProtocol.enable=true`。
- `assetProtocol.scope` 至少覆盖应用图片缓存所在目录，即 `$APPDATA/images/**`，并同时覆盖 `$APPLOCALDATA/images/**`，避免不同平台目录映射差异导致图片资源不可见。
- 将该能力纳入发布配置校验；若未来有人移除 `assetProtocol` 或缩小 `scope`，自动化校验必须直接失败。

### 影响

- 图片缩略图与原图在真实桌面环境中能够被 `WebView` 正常读取，`convertFileSrc` 才真正具备可用性。
- 这是一条运行时配置门禁，不改变数据库结构、IPC 契约和前端图片回退逻辑，但会直接影响图片记录是否可见。
- 后续若图片存储目录迁移，必须同步更新 `assetProtocol.scope` 与发布校验脚本。

## 2026-03-09：图片预览统一采用 `convertFileSrc` 解析本地资源，并只对已挂载卡片懒加载原图回退

### 背景

- 当前图片记录摘要里保存的是本地文件路径，若前端直接把 `thumbnail_path` 塞给 `<img>`，在 `Tauri WebView` 中会被当作普通页面相对路径，导致“文件明明存在但预览不显示”。
- `v1.2` 同时要求补齐 `thumbnail -> original -> placeholder` 回退链路，但若对整个列表批量请求 `get_record_detail`，会把原图回退成本扩散到所有历史记录，偏离“只对可见卡片按需回退”的既定方向。

### 决策

- 图片卡片渲染缩略图与原图时，统一先通过 `convertFileSrc` 把本地文件路径转换为 WebView 可访问资源 URL。
- 原图回退不在主面板列表初始化时批量拉取，而是在图片卡片已挂载、缩略图不可用或加载失败时，再按需调用 `get_record_detail` 读取 `image_detail.original_path`。

### 影响

- 图片预览问题从根因上收敛：既解决了路径不可显示的问题，也补齐了失败时的原图与占位态退化逻辑。
- 原图回退成本被限制在当前已挂载的图片卡片范围内，不会因为长列表一次性放大详情读取压力。
- 该决策只影响前端图片渲染与详情读取时机，不新增 IPC 命令，也不改变数据库与事件契约。

## 2026-03-09：主面板鼠标双击粘贴与键盘粘贴统一走共享执行链路，并以前端防重入锁避免重复提交

### 背景

- `v1.2` 明确要求主面板补齐“单击选中 + 双击粘贴”，而现有实现里真正的粘贴执行逻辑只存在于键盘事件处理，鼠标链路如果单独再实现一套，会让权限校验、失败提示、成功隐藏面板等行为继续分叉。

- 权限不足或粘贴失败时，主面板保持当前可见状态，仅展示 toast 与权限引导，不额外做隐藏回退。

- 鼠标与键盘操作的行为一致性更高，后续继续扩展卡片级交互时不需要维护两套粘贴状态机。
- 双击重复提交的风险被控制在前端入口层，能更稳定地满足 `BDD-NFR-21` 的验收要求。
- 该决策只影响前端运行态与交互逻辑，不改变现有 IPC 契约、数据库结构与配置模型。

- `macOS` 下主面板覆盖 `Dock` 的能力由“坐标策略 + 原生层级策略”共同保证，问题边界更清晰。
- 后续若继续出现 `Dock` 遮挡问题，可以优先排查窗口层级与显示流程，而不是重新怀疑定位坐标。
- 平台差异被限制在 `window` 模块内部，不影响 IPC、前端状态同步与快捷键语义。

## 2026-03-08：v1.1 主面板改为“失焦自动隐藏 + 覆盖 Dock + `Command+数字` 快贴”交互模型

### 背景

- `v1.0` 为避免主面板被 `macOS Dock` 遮挡，曾将面板定位与尺寸计算建立在 `visibleFrame / work_area` 之上；但这会直接缩短主面板有效高度，导致底部内容被裁剪，问题本质没有解决。
- 当前前端还存在把窗口 `focus` 当作“自动重新显示主面板”信号的实现倾向，这与“桌面浮层应在失焦后自动收起”的产品预期冲突，容易造成前后端显隐状态错位。
- 现有数字键能力只能做到“快选”，高频用户仍需要额外按一次 `Enter` 才能粘贴，键盘链路仍有优化空间。

### 决策

- `v1.1` 起，`macOS` 主面板不再通过缩短高度或基于 `visibleFrame / work_area` 避让 `Dock`；改为基于目标显示器的物理屏幕 frame 贴底定位，恢复设计高度 `220px`，并允许主面板直接覆盖 `Dock`。
- 主面板显隐以原生窗口状态为最终真相来源；当主面板失去焦点时自动隐藏，并向前端广播显隐变化，前端不再依赖浏览器窗口 `focus` 事件强行恢复可见状态。
- 在主面板可见时，保留 `1~9` 仅快选的既有语义，同时新增 `Command + 1~9` 直接粘贴当前顺序内对应记录；左右切换和数字快选若命中视口外卡片，必须自动滚动到可见区域。
- 当前版本只对 `macOS` 定义 `Command + 数字` 快贴，不同步扩展为 `Windows / Linux` 的 `Ctrl + 数字`，避免在同一版本内扩大快捷键冲突面。

### 影响

- 主面板底部内容不再因为“规避 `Dock`”而被裁剪，窗口行为更符合桌面浮层产品直觉。
- 主面板的显示、隐藏、快贴、自动滚动四条高频链路将围绕同一套运行态语义收敛，减少前后端状态错位和行为分叉。
- 本轮改动集中在窗口层、键盘层和前端视图层，不新增数据库表、不新增配置字段，风险面相对可控。

## 2026-03-07：macOS 多显示器面板定位改为直接使用 `NSScreen.visibleFrame` 原生坐标设窗

### 背景

- 首轮修复后，副屏首次显示已经正确，但主屏 / 副屏反复切换显示与隐藏时，主面板在主屏仍会偶发飞到屏幕中央。
- 最新手测日志出现 `display_id=1 x=6880 width=2880 height=5070` 这类异常值，明显比目标可见区域大一倍，说明“选对屏”后仍存在坐标和缩放重复换算。
- `tauri / tao` 在 macOS 下处理 `set_position / set_size` 的物理坐标时，会结合当前窗口的 `scale_factor` 做转换；当窗口刚停留在 1x 副屏、下一次又要移动到 2x 主屏时，目标坐标容易被按错误缩放解释。

### 决策

- macOS 下继续使用 CoreGraphics `CGGetDisplaysWithPoint` 负责系统级选屏。
- 主面板目标位置与尺寸直接读取目标 `NSScreen.visibleFrame`，按 Cocoa 原生点坐标构造贴底 panel frame。
- macOS 下窗口移动与缩放直接对原生 `NSWindow` 调用 `setFrame:display:animate:`，不再走 Tauri 的物理坐标 `set_position / set_size` 路径。

### 影响

- 主屏 / 副屏 / 混合缩放来回切换时，不再依赖“当前窗口缩放因子”和“目标显示器缩放因子”的组合正确性，位置错乱概率显著降低。
- 调试日志会同时输出目标 frame 与实际 frame，便于后续继续追查极端桌面布局问题。
- 非 macOS 平台仍保持现有通用定位流程，不扩大平台差异面。

## 2026-03-07：macOS 多显示器定位改为直接走 CoreGraphics 选屏，不再依赖 Tauri 光标坐标

### 背景

- 手动测试发现：主面板在多显示器场景下仍然稳定落在主显示器，说明当前“根据鼠标所在屏幕选中目标显示器”的实现未真正生效。
- 现有实现依赖 `tauri::AppHandle::cursor_position()` 再和 `available_monitors()` 的 `work_area` 做命中判断，但 macOS 下该光标坐标链路来自 `tao`，会混入主屏缩放因子，导致混合缩放显示器场景下坐标失真。

### 决策

- macOS 下主面板选屏改为直接调用 CoreGraphics `CGGetDisplaysWithPoint`，以系统级显示器 ID 判断鼠标当前所在显示器。
- 目标显示器的可用工作区不再依赖 Tauri 反推，而是基于 `CGDisplayBounds + NSScreen.visibleFrame + backingScaleFactor` 直接计算。
- 非 macOS 平台仍保留原有 `cursor_position + available_monitors` 通用逻辑，避免扩大改动面。

### 影响

- 混合缩放或副屏场景下，主面板更有机会准确跟随当前鼠标所在显示器，而不是错误回退到主显示器。
- 定位实现会多一层 macOS 平台分支，但根因被收敛在窗口模块内部，不影响 IPC 与前端契约。
- 后续若 Tauri / tao 修复光标坐标问题，也可以在当前隔离层内切换回官方实现。

# 项目决策记录文档

## 2026-03-06：v0.2 持久化采用 SQLite 三表 + 图片资源文件落盘方案

### 背景

- `v0.2` 需要在 `v0.1` 文本内存历史的基础上新增图片、文件/目录记录，并支持应用重启后恢复历史。
- 图片原始二进制若直接写入 SQLite，会放大数据库体积、增加备份与读取压力，也不利于缩略图的独立清理。
- 文件记录本质上是“一个历史记录包含多个路径项”，需要与主记录建立 1:N 关系。

### 决策

- 历史记录统一采用 `clipboard_items` 主表保存公共字段。
- 图片类型使用 `image_assets` 表保存元数据，原图与缩略图文件存放到应用数据目录。
- 文件类型使用 `file_items` 表保存路径明细，通过外键与主表关联。
- 数据库迁移版本使用 `PRAGMA user_version` 管理，不新增业务外的版本表。

### 影响

- 主面板列表查询可通过主表 + 子表 Join 输出轻量摘要，便于后续扩展。
- 图片记录删除时需要执行“数据库删除 + 文件清理”的组合流程，但可通过异步补偿保证最终一致性。
- 为后续 `v0.3+` 的搜索、筛选、分页提供可扩展数据底座。

## 2026-03-06：v0.2 主界面继续保持单页模型，摘要与详情分离

### 背景

- 当前版本目标聚焦在“复制后快速找回并再次粘贴”，而不是扩展更多管理界面。
- 图片原图路径、文件明细列表等详情数据体积明显大于主面板卡片渲染所需信息，若一次性全部返回会拖慢面板打开速度。

### 决策

- `v0.2` 继续仅保留一个核心页面：主面板，不引入设置页、托盘页、搜索页。
- `get_records` 只返回卡片摘要 DTO，满足主面板首屏渲染。
- 新增 `get_record_detail` 获取完整详情，供粘贴链路和后续扩展按需读取。
- 记录更新采用事件推送，缩略图就绪、复用置顶等场景通过 `clipboard:record-updated` 通知前端增量刷新。

### 影响

- 主面板打开时延可控，图片和文件详情不会拖慢首屏。
- 前端状态模型从“单一文本记录列表”升级为“摘要列表 + 按需详情读取”。
- 后续若引入搜索和详情面板，可复用现有摘要/详情分离契约。

## 2026-03-06：v0.2 默认保留上限设为文本 200、图片 50、文件 100，纯文本粘贴仅限文本

### 背景

- 持久化引入后，历史记录不再随着应用退出自动清空，需要明确容量边界，避免数据库与图片资源无限增长。
- 文本、图片、文件三类内容的体积差异显著，不能复用同一个固定上限。
- `Shift + Enter` 的纯文本模式仅对文本语义成立，对图片和文件没有合理的降级结果。

### 决策

- 默认保留上限设置为：文本 `200`、图片 `50`、文件 `100`。
- 清理策略按内容类型分别执行，仅淘汰对应类型的最旧记录。
- `plain_text` 粘贴模式仅允许 `text` 类型记录使用；前端默认禁用非文本场景，后端使用 `INVALID_PARAM` 兜底。

### 影响

- 持久化空间占用更可控，图片记录数量被重点限制。
- 用户在主面板中的历史结构更稳定，不会因某一类型高频复制挤占所有其他类型。
- API、BDD 与状态码契约可围绕统一的模式限制达成一致。

## 2026-03-06：macOS 粘贴模拟增加焦点稳定等待与按键驻留时长

### 背景

- 在 macOS 自动化验收中发现：历史记录通过 `Enter` 触发粘贴后，系统剪贴板内容已正确更新，但焦点刚从面板切回目标应用时，`Cmd+V` 模拟偶发丢失。
- 现有实现中，面板隐藏后仅等待 `80ms` 就立即发送 `Cmd+V`，并且按键按下与抬起之间没有任何驻留时间，对焦点切换较慢的应用不够稳健。

### 决策

- 将面板隐藏后的“焦点稳定等待”延长到 `140ms`，给目标应用更多时间完成前台激活。
- macOS 按键模拟补充 `20ms` 的按下驻留时长，再发送抬起事件，降低目标应用漏收组合键的概率。

### 影响

- 历史记录粘贴在 TextEdit、Terminal 等需要前台焦点的应用中稳定性更高。
- 粘贴总耗时会略微增加，但仍保持在 v0.1 的交互预算范围内；后续可在 v0.2 E2E 中继续精细量化。

## 2026-03-06：主面板定位改为基于可用工作区，并提升 macOS 原生窗口层级

### 背景

- 手动测试发现：主面板虽然已经开启 `alwaysOnTop`，但在 macOS 下仍会被底部 Dock 遮挡。
- 现有窗口定位逻辑直接使用整块屏幕高度贴底，未扣除 Dock 和菜单栏占用的 `work_area`。
- Tauri 的 `alwaysOnTop` 在 macOS 对应 `NSFloatingWindowLevel`，层级低于 Dock 的 `NSDockWindowLevel`，因此仅靠现有配置不足以保证面板始终浮在系统 UI 之上。

### 决策

- 主面板的宽度、横向起点和纵向位置统一基于显示器 `work_area` 计算，确保不会落入 Dock 或菜单栏占用区域。
- macOS 下在显示主面板时，额外将原生窗口层级提升到 `NSStatusWindowLevel`，高于 Dock。
- 主面板在 macOS 下加入“所有空间可见”与“全屏辅助窗口”行为，保证从不同 Space 或全屏应用唤起时仍能显示在最前。

### 影响

- 面板显示时会紧贴可用桌面区域底部，而不是紧贴物理屏幕最底部。
- Dock 位于底部、左侧或右侧时，面板都会自动避让，不再被系统栏遮挡。
- 从普通桌面、Mission Control 多空间或全屏应用中唤起面板时，显示一致性更高。

## 2026-03-06：重复内容统一复用原记录，粘贴成功后显式置顶

### 背景

- 手动验收发现：当用户从历史卡片中执行粘贴时，监听模块会把这次“程序写入系统粘贴板”的内容再次捕获，导致列表新增一条重复记录。
- 现有 v0.1 设计仅对“最新一条记录”做去重，旧记录再次进入系统粘贴板时仍会生成新 ID，和“历史列表保持简洁”的目标不一致。

### 决策

- 文本去重从“仅比较最新记录”调整为“对整条历史列表做精确匹配”。
- 如果新文本与非最新记录相同，不再新建记录，而是复用原记录并移动到最左侧。
- 用户执行粘贴时，不再依赖监听器误捕获来触发置顶；改为在粘贴成功后由后端显式将对应记录置顶。
- 粘贴流程在写入系统粘贴板后，先同步监听器快照，再恢复监听，避免把应用自身写入再次识别成新变化。

### 影响

- 粘贴历史记录后，列表总数保持不变，但被粘贴的记录会移动到最左侧。
- 用户再次复制一段历史中已存在但当前不是最新的文本时，也会复用原记录并置顶，不再出现重复卡片。
- `paste_record` IPC 返回被置顶后的 `ClipboardRecord`，前端可立即同步本地状态。

## 2026-03-06：`config.json` 采用扁平字段结构，并在启动时自动补齐缺失上限

### 背景

- `v0.2` 需要在原有文本上限之外，再新增图片与文件两类记录上限。
- `v0.1` 的配置模型只有 `max_text_records` 与 `toggle_shortcut`，如果直接切换到严格新结构，旧用户升级后会面临缺字段启动失败或需要手工重建配置的问题。
- 当前版本尚未引入设置页，用户对容量上限的观察与调整仍主要依赖运行时目录下的 `config.json`。

### 决策

- `config.json` 继续采用扁平字段结构：`max_text_records`、`max_image_records`、`max_file_records`、`toggle_shortcut`。
- 应用启动时从运行时目录加载 `config.json`；若文件不存在，则自动创建一份默认配置。
- 读取旧配置时，对缺失的新字段自动回填默认值，并将补齐后的完整配置回写到文件，保证用户能直接看到三类上限配置。

### 影响

- `v0.1` 升级到 `v0.2` 不需要手工删除或重建配置文件，升级路径更平滑。
- 用户可以直接在同一份配置文件中看到文本、图片、文件三类保留上限。
- 后续若补充设置页或配置编辑能力，可以沿用这套配置字段与兼容迁移策略。

## 2026-03-07：v0.5 将 `config.json` 从扁平结构升级为分组化 `config_version=2`，黑名单规则继续落在配置层

### 背景

- `v0.3` 之前的 `config.json` 仅包含少量扁平字段，适合没有设置页的阶段；进入 `v0.5` 后，配置项将扩展到主题、容量、快捷键、黑名单规则等多个分组，继续堆叠扁平字段会让语义和迁移成本迅速上升。
- 黑名单规则本质上是用户偏好，而不是业务历史；若为了设置页引入新的 SQLite 业务表，会把历史数据和配置规则耦合在一起，增加迁移与排查成本。
- 现有 SQLite 三表结构已经足以支撑文本、图片、文件历史记录，不需要为了 `v0.5` 的设置能力再扩表。

### 决策

- `config.json` 在 `v0.5` 升级为分组化结构：`general`、`history`、`shortcut`、`privacy`，并增加 `config_version=2`。
- `v0.3` 及更早版本的扁平配置在启动时自动迁移到新结构；旧值保留，新增字段回填默认值，并将迁移结果回写到同一份 `config.json`。
- 黑名单规则继续作为配置数据，落在 `privacy.blacklist_rules` 中，不新增 SQLite 业务表。
- `clipboard.db` 继续只承载历史记录数据；主题、快捷键、黑名单、自启动等设置项统一落在配置层或系统集成产物中。

### 影响

- 设置窗口可以直接按领域分组读写配置，前后端契约更稳定，也更接近用户心智模型。
- 升级到 `v0.5` 时不需要重建数据库，只需执行一次轻量配置迁移，升级路径更平滑。
- 后续如果增加导入导出或设置备份，配置文件可以整体搬运，不需要跨 SQLite 和 JSON 做双向拼装。

### 影响

- 后续设置 IPC、设置页表单和监听过滤链路都可以共享同一套领域规则，减少重复实现和漂移风险。

- 设置校验错误可以稳定映射到 `SETTINGS_VALIDATION_ERROR` / `BLACKLIST_RULE_DUPLICATE` 等契约，便于 UI 做字段级反馈。

## 2026-03-07：平台能力统一由 `PlatformCapabilityResolver` 输出，Wayland 与未知 Linux 会话采用保守降级

### 影响

- 后续设置页、托盘和运行态查询都可以共享同一份能力矩阵，减少平台状态不一致问题。

- 后续 Windows / Linux 真正接入平台 API 时，可以在不改动上层契约的前提下逐步替换底层实现。

## 2026-03-07：黑名单过滤在监听阶段按活动应用最小标识匹配，检测失败默认放行

### 背景

- `TASK-03-4` 已让黑名单规则可以被配置，但在 `TASK-04-3` 之前监听链路还不会真正消费这些规则，隐私能力仍停留在 UI 层。
- 平台能力矩阵已经明确：macOS、Windows、Linux X11 具备活动应用识别能力，Wayland 与未知 Linux 会话则不稳定或不可用，过滤链路必须能随能力降级。
- 如果把黑名单判断放到入库之后再补偿删除，敏感内容仍会短暂进入运行态仓库甚至 SQLite，不符合 `v0.5` 的隐私目标。

### 决策

- 黑名单判断放在 `ClipboardMonitorService` 中执行：完成剪贴板快照去重后、调用运行态仓库前读取当前活动应用并做规则匹配，命中即直接跳过本次采集。
- 平台活动应用识别统一走 `PlatformActiveAppDetector`：macOS 读取前台应用 `bundle_id`，Windows 读取前台进程可执行名并同时作为 `process_name / app_id` 的最小标识，Linux X11 读取 `WM_CLASS` 与进程名。

- 若当前会话不支持活动应用识别，或本次检测命令失败，则记录告警并放行采集，不让黑名单能力失效反向阻塞主流程。

### 影响

- 隐私能力已经从“可配置”升级为“监听阶段真实生效”，敏感应用中的复制内容不会再进入历史数据库。
- 平台能力与黑名单过滤形成统一降级模型：支持则执行过滤，不支持则明确受限但保持复制主流程可用。

## 2026-03-08：图片原图缓存采用共享 `LRU` 双阈值策略，最近历史摘要在 SQL 层直接计算文本元数据

### 背景

- `TASK-03-1` 已把主面板历史记录读取上限提升到 `200`，接下来瓶颈从“渲染数量”转向“图片恢复链路和摘要查询负载”。
- 当前原图粘贴每次都会重新打开磁盘文件并解码 `PNG`，高频粘贴同一批截图时会重复消耗 IO 与 CPU。
- 最近历史摘要查询此前会把整段 `text_content` 带回 Rust 再计算字数与行数，长文本场景会产生不必要的数据搬运。

### 决策

- `ImageStorageService` 为原图读取链路新增共享内存 `LRU` 缓存，缓存上限固定为“最多 `32` 张原图 + 最多 `64MB` 解码后字节”，任一阈值超出即按最近最少使用策略淘汰。
- 原图缓存只在真正读取原图时填充；缩略图浏览与图片落盘阶段不让原图常驻内存，保持“按需加载”的发布策略。
- 删除图片资产时同步驱逐对应缓存项，确保磁盘与内存生命周期一致，不给后续孤立文件清理留下隐患。
- 最近历史摘要查询改为由 `SQLite` 直接计算 `text_char_count / text_line_count`，Rust 侧只接收元数据，不再搬运完整正文。

### 影响

- 高频图片粘贴与预览场景会减少重复解码，对长时间运行时的 CPU / IO 抖动更友好。
- 原图缓存具备确定的内存预算，不会因为重复访问截图历史而无限增长。
- 主面板大列表查询在文本记录很多时会更轻，后续 `TASK-03-3` 和 `TASK-04-1` 可以继续围绕真实负载补齐治理与回归。

## 2026-03-08：孤立图片清理统一复用 `ImageCleanupService`，同时支持关于页手动触发与固定周期后台执行

### 背景

- `v1.0` 已经具备孤立图片扫描基础，但在 `TASK-03-3` 之前还缺少“真正删除文件、回写最近清理统计、前端手动触发入口”的闭环。
- 如果把手动清理和后台定时清理拆成两套逻辑，容易出现删除规则、日志字段和诊断摘要不一致。
- 最近一次清理结果属于当前运行态健康信息，不适合写入 `SQLite` 或 `config.json`，否则会把短期运维状态误当成持久业务数据。

### 决策

- 新增 `ImageCleanupService` 作为唯一清理执行入口，内部负责：目录扫描、数据库引用比对、孤立文件删除、统计汇总与日志落盘。
- 关于页通过 `run_orphan_cleanup` IPC 手动触发一次清理；应用启动后同时启动固定周期后台清理任务，每 `6` 小时执行一次，二者复用同一服务。
- 最近一次清理摘要仅保存在运行态内存中，并通过 `DiagnosticsSnapshot.last_orphan_cleanup` 与 `system:diagnostics-updated` 事件向前端回写。
- 清理过程只删除数据库已无引用的文件；单个文件删除失败仅记录告警并继续执行，不让一次权限或文件系统异常阻断整轮治理。

### 影响

- 孤立图片治理从“能扫描”升级为“能持续收敛并可人工介入”，磁盘残留不会随着长期使用持续累积。
- 关于页现在不仅能查看诊断，还能主动执行低风险治理操作，并立即看到结果回显。
- 后续 `TASK-04-1` 可以直接围绕同一条清理链路补齐更多集成与 BDD 回归，不必分别维护手动和后台两套行为模型。

## 2026-03-08：v1.0 发布链路采用“日常 CI 无签名冒烟 + tag 触发正式签名发布”，macOS `DMG` 改为无界面脚本封装

### 背景

- `TASK-04-2` 需要同时收口三平台打包、签名、发布清单与回滚说明，但当前仓库只有单平台 `pnpm tauri build`，缺少正式发布工作流与发布前校验入口。
- Tauri 默认 `DMG` 生成依赖 `create-dmg` 的 Finder AppleScript，美化安装盘界面时在终端 / CI 环境中容易阻塞，不适合作为稳定的自动化发布基线。
- 日常分支验证与正式发布对签名要求不同：CI 需要尽快发现跨平台构建回归，而正式发布才需要导入证书、执行签名与上传 Release 产物。

### 决策

- 日常 `CI` 采用三平台 `--no-sign` 打包冒烟，只验证能否生成当前平台 bundle；正式发布通过 tag 触发独立 `release.yml` 工作流，执行签名、产物收集和 Draft Release 上传。
- Windows 通过 Tauri `signCommand` 调用仓库内的 `scripts/sign-windows.ps1`，从 GitHub Secrets 导入 `.pfx` 后使用 `signtool` 完成签名。
- macOS 保留 Tauri 对 `.app` 的签名 / notarization 流程，但 `DMG` 改由 `scripts/package-macos-dmg.sh` 使用 `hdiutil` 无界面封装，避免 Finder AppleScript 造成构建不稳定。
- 发布前统一执行 `release:verify-config`、`release:check` 与 `release:build:local`，把版本校验、全量自动化回归与本机打包冒烟收敛为标准发布门禁。

### 影响

- `v1.0` 发布链路具备更清晰的职责分层：日常 CI 更快，正式发布更可控。
- macOS `DMG` 打包在本地终端与 CI 环境中的成功率更高，减少因为 GUI 自动化失败导致的发布阻塞。
- 后续版本只需维护同一套发布清单与 Secrets 规范，即可复用现有 tag 发布流程。

## 2026-03-09：v1.3 范围锁定为“空格聚焦预览 + 右键上下文菜单 + 半透明科技风视觉升级”

### 背景

- `v1.2` 已经补齐主面板的鼠标选中、双击粘贴、图片预览回退与可视快捷编号，但用户仍然缺少“在粘贴前快速确认完整内容”的顺滑入口。
- 当前主面板虽然功能完整，但鼠标用户仍然主要依赖双击与少量按钮，缺少符合桌面习惯的右键上下文菜单。
- 用户明确希望 `v1.3` 在视觉上更好看、更流行、更有科技感，因此本轮需要把风格升级纳入版本主目标，而不是只做交互补丁。

### 决策

- `v1.3` 只聚焦 3 项能力：空格聚焦预览、右键上下文菜单、主面板半透明科技风视觉升级。
- 当前版本不扩展搜索、标签、收藏、批量操作、OCR、同步等新业务能力。
- 当前版本不新增 SQLite 业务表，不升级配置结构，不开放用户自定义外观参数。

### 影响

- 版本目标从“功能补洞”转向“交互效率 + 视觉质感”的集中升级，范围更明确。
- 后续文档、测试、页面说明与冲刺任务都会围绕这 3 条主线展开，避免版本再次发散。
- `v1.3` 的实现重心将主要落在前端运行态与样式系统，而不是后端能力扩张。

## 2026-03-09：v1.3 的预览层与右键菜单复用既有详情 / 粘贴 / 删除命令，不新增后端 IPC

### 背景

- 空格预览与右键菜单都属于新的交互入口，如果每个入口都单独定义新命令，会让前后端契约面快速膨胀。
- 当前后端已经具备 `get_record_detail`、`paste_record`、`delete_record`、`hide_panel` 等能力，理论上足以支撑 `v1.3` 的主要操作。
- 本轮版本更强调体验迭代，如果同时扩张 IPC 面与存储结构，会增加实现与回归成本。

### 决策

- 聚焦预览统一通过既有 `get_record_detail` 按需拉取完整内容。
- 右键菜单中的 `直接粘贴`、`纯文本粘贴`、`删除记录` 分别复用既有 `paste_record`、`delete_record` 与 `hide_panel` 链路。
- 预览打开状态、菜单锚点、菜单动作禁用态等全部停留在前端运行态，不进入后端或数据库。

### 影响

- `v1.3` 的实现风险主要集中在前端状态编排、层级管理与组件拆分，后端改动面较小。
- 自动化测试可以更聚焦于前端交互闭环与对既有命令的复用，而不是扩展新的命令契约。
- 当前版本仍能维持较稳定的存储与命令边界，为后续迭代保留空间。

## 2026-03-09：主面板视觉采用“玻璃拟态 + 科技强调色”的默认样式方案，本版不开放个性化外观配置

### 背景

- 用户对 `v1.3` 的明确要求是“更好看一些、更流行、更科技”，说明当前界面在视觉质感层面还有明显提升空间。
- 若直接开放透明度、模糊度、色彩、动效强度等个性化配置，会显著增加设计与测试组合复杂度。
- 当前更适合先收敛一套高质量默认视觉语言，再根据真实反馈评估是否需要开放可配置主题。

### 决策

- `v1.3` 主面板默认采用半透明背景、弱模糊、细边框、柔和阴影、青蓝 / 蓝紫系科技强调色。
- 卡片统一建立默认、悬停、选中、预览中 4 套核心视觉状态，强化层级而不过度装饰。
- 当前版本不新增 `appearance` 配置项，不开放模糊度、透明度、光效强度等用户偏好设置。

### 影响

- 设计与开发可以集中打磨一套统一视觉方案，避免一开始就陷入多主题分支维护。
- 视觉回归测试将成为 `v1.3` 的重点，尤其需要关注深 / 浅主题和不同平台上的可读性表现。
- 若后续用户对视觉偏好反馈明确，可在后续版本单独规划“外观设置”能力。
