# v1.5 技术可行性评审文档

## 1. 评审概述

### 1.1 评审目的

本文档用于评审 `v1.5` 新增能力的技术可行性，重点判断以下目标是否可以在当前 `Tauri v2 + React 19 + TypeScript + Rust + SQLite` 架构下稳定落地：

- 复制 / 粘贴 / 显示预览三类音效反馈
- 预览窗口打开后的自动联动切换
- 音频与视频预览播放
- PDF、DOC / DOCX、EXCEL、PPT 文稿预览
- 超链接内容预览

### 1.2 评审范围

- 音效资源打包、命名与播放方式
- 预览窗口状态同步与窗口复用策略
- 媒体内容本地播放可行性
- 文稿解析、内容抽取与预览缓存可行性
- 链接抓取、摘要生成与安全降级可行性

### 1.3 评审方法

- 结合项目现有实现现状做增量评估
- 对每个能力给出推荐方案、可行性等级、风险点与降级方案
- 优先选择“可迭代交付”的方案，不追求一步到位的高保真渲染

### 1.4 评审结论

**总体评估**：✅ 可行  
`v1.5` 所有目标能力在当前技术栈下都可以实现，但应采用“分层预览 + 异步准备 + 失败降级”的策略，其中音效与媒体播放为高可行项，PDF 预览为中高可行项，Office 文稿与超链接预览属于中等可行项，需要明确接受“结构化内容预览优先于像素级还原”的版本结论。

## 2. 能力可行性总览

| 能力 | 推荐方案 | 可行性 | 主要风险 |
|------|----------|--------|----------|
| 复制 / 粘贴 / 预览音效 | 后端统一音效服务 + 项目内英文资源文件 | 高 | 隐藏窗口场景下的播放稳定性 |
| 预览自动联动 | 复用现有预览窗口 + 扩展目标记录切换语义 | 高 | 频繁切换时的抖动与重复加载 |
| 音频预览 | WebView 原生音频控件 + 本地文件源适配 | 高 | 本地路径不可访问 |
| 视频预览 | WebView 原生视频控件 + 封面 / 首帧策略 | 高 | 编码兼容性与大文件首帧加载 |
| PDF 预览 | PDF 专用渲染器 + 分页滚动查看 | 中高 | 大文件加载与缓存体积 |
| DOC / DOCX 预览 | 优先结构化内容抽取，DOC 允许降级 | 中 | 旧二进制格式兼容差异 |
| XLS / XLSX 预览 | 表格数据抽取 + 工作表切换 | 中 | 超大表格渲染性能 |
| PPT / PPTX 预览 | 幻灯片标题 / 文本抽取 + 页导航 | 中 | 排版高保真难度高 |
| 超链接预览 | 后端抓取标题、摘要、主图与正文片段 | 中 | 站点反爬、超时、X-Frame-Options |

## 3. 音效反馈可行性评审

### 3.1 候选方案对比

| 方案 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| 前端 `HTMLAudioElement` 播放 | 实现简单，前端易调试 | 面板隐藏或窗口未激活时触发链路不稳定 | 不推荐作为主方案 |
| 后端统一音效服务播放 | 与复制 / 粘贴 / 预览成功事件天然同层，窗口不可见也可触发 | 需要新增资源定位与播放封装 | 推荐 |
| 直接调用系统默认提示音 | 实现简单 | 无法满足用户指定音效文件要求 | 不满足需求 |

### 3.2 推荐结论

- 建议把 `docs/音效/` 中的 3 个资源复制到项目内，并改为英文命名：
  - `copy-notification.mp3`
  - `paste-notification.mp3`
  - `preview-open.mp3`
- 音效播放库建议采用 `Howler.js`，利用其统一 API、自动缓存与 `Web Audio / HTML5 Audio` 回退能力承接短音效播放
- 播放职责建议落在统一 `SoundEffectService`，由复制入库成功、粘贴成功、预览内容首次显示成功等事件驱动
- 音效失败时只记录日志，不反向中断主流程

### 3.3 可行性结论

✅ **高可行**，实现复杂度低，收益明显，建议优先落地。

## 4. 预览联动可行性评审

### 4.1 现状判断

- 当前项目已经有独立预览窗口、`show_preview_window` 命令和 `system:preview-window-requested` 事件
- 预览窗口当前已经支持在同一窗口中接收新的 `record_id` 并重新加载详情
- 因此 `v1.5` 的重点不是新建预览机制，而是把“窗口可见时切换选中卡片”接到现有事件链路上

### 4.2 推荐方案

- 扩展 `show_preview_window(recordId)` 的语义：当预览窗口已经存在时，只切换目标记录，不重新创建窗口
- 主面板在“预览窗口可见”的前提下监听选中索引变化，主动派发新的预览目标
- 预览窗口内部继续复用 `useRecordPreviewDetail(recordId)` 拉取详情

### 4.3 风险与缓解

- 风险：用户快速按住上下键时会造成频繁请求  
  缓解：允许轻量节流，只保留最后一次目标记录
- 风险：切换到损坏记录时窗口残留旧内容  
  缓解：切换前先进入 loading 态，失败后进入明确 error / fallback 态

### 4.4 可行性结论

✅ **高可行**，属于既有机制的语义扩展，不需要新建窗口系统。

## 5. 音频与视频预览可行性评审

### 5.1 推荐方案

- 继续沿用当前本地文件预览地址转换方案，把文件路径转换为 WebView 可读的本地资源地址
- 媒体预览统一采用 `Vidstack` 作为播放器层，承接 `React` 集成、控件状态、音频 / 视频统一外壳与后续扩展能力
- 音频预览至少支持播放、暂停、进度拖动、基础元信息展示
- 视频预览至少支持播放、暂停、进度拖动、封面或首帧显示
- 如果未来出现 `m3u8` 一类流媒体来源，可在 `Vidstack` 之外按需叠加 `hls.js`；当前版本不作为必选依赖
- 若后续需要波形可视化，可单独评估 `wavesurfer.js`，但不纳入 `v1.5` 基线

### 5.2 风险与缓解

| 风险 | 说明 | 缓解 |
|------|------|------|
| 编码不兼容 | 少量视频编码在系统 WebView 中可能不可播，播放器库无法替代底层编解码支持 | 显示“不支持当前编码预览”，保留文件元信息 |
| 路径访问失败 | 文件被移动或权限不足 | 进入降级态，允许用户看到路径与错误提示 |
| 大文件首帧慢 | 视频封面获取较慢 | 先显示 loading，再切换播放器 |

### 5.3 可行性结论

✅ **高可行**，当前技术栈对本地媒体播放支持充分。

## 6. PDF 与 Office 文稿预览可行性评审

### 6.1 推荐策略

- `PDF`：优先采用 `PDF.js` 一类专用 PDF 渲染方案，支持分页滚动和页数展示
- `DOCX`：优先采用 `Mammoth.js` 或 `docx-preview` 这类开源库，把正文转换为可读 HTML 或结构化内容
- `XLSX`：优先采用 `SheetJS` 这类开源库，把工作表与表格内容转换为结构化预览
- `PPTX`：优先采用基于 OOXML 解包与文本抽取的开源解析方案，先满足标题、正文与页级结构预览
- `DOC / XLS / PPT`：当前版本不引入 `LibreOffice` 或系统级 Office 转换依赖；旧二进制格式优先尝试轻量抽取，失败则退化为“不可预览 + 文件信息”

### 6.2 为什么不追求像素级还原

- 当前项目核心是“剪贴板确认与快速浏览”，不是办公套件
- 高保真还原会显著放大跨平台兼容、第三方依赖与包体积压力
- 用户当前需求是“显示文稿内容”，结构化内容预览已能满足多数确认场景
- 引入 `LibreOffice` 会增加运行时依赖、跨平台安装成本、冷启动耗时与缓存清理复杂度，不符合当前版本追求的轻量交付目标

### 6.3 风险与缓解

| 风险 | 说明 | 缓解 |
|------|------|------|
| 旧 Office 二进制格式兼容差 | `DOC / XLS / PPT` 解析稳定性低于 OOXML | 优先支持 `DOCX / XLSX / PPTX`，旧格式允许降级 |
| 大文稿抽取慢 | 大 PDF 或多工作表 Excel 解析耗时高 | 采用异步准备与缓存，先显示 loading |
| 表格 / 幻灯片过大 | 直接全量渲染会卡顿 | 默认首屏只显示前若干行 / 页，并允许分段查看 |
| 第三方库输出 HTML 安全性 | `DOCX` 转 HTML 场景可能带来不可信内容渲染风险 | 统一做 HTML 清洗与白名单渲染 |

### 6.4 可行性结论

✅ **中高可行**，但必须接受分层支持策略：`PDF` 优先级最高，`OOXML` 文档次之，旧二进制格式允许降级；当前版本明确**不引入 `LibreOffice`** 作为预览依赖。

## 7. 超链接预览可行性评审

### 7.1 推荐方案

- 后端根据 `primary_uri` 发起安全抓取
- 提取标题、站点名、摘要、主图、正文片段等预览友好字段
- 前端渲染为“链接卡片 + 摘要内容 + 外部打开按钮”组合视图

### 7.2 不推荐的方案

- 不建议直接在预览窗口里用 `iframe` 载入任意第三方网页作为主方案
- 原因是大量站点会设置 `X-Frame-Options`、`CSP` 或登录态限制，稳定性差且安全成本高

### 7.3 风险与缓解

| 风险 | 说明 | 缓解 |
|------|------|------|
| 抓取超时 | 远端站点响应慢 | 设置超时并进入降级态 |
| 页面不可抓取 | 反爬、权限、动态渲染页面 | 至少展示 URL、站点名和失败说明 |
| 内容不安全 | 原始 HTML 含脚本 | 后端先抽取安全摘要，再传给前端 |

### 7.4 可行性结论

✅ **中等可行**，推荐做“摘要型链接预览”，不做“完整网页浏览器”。

## 8. 数据与缓存可行性评审

### 8.1 推荐结论

- `v1.5` 建议升级到新的 schema 版本，新增预览状态字段与预览资源表
- 文稿、链接与封面类资源建议落磁盘缓存，数据库只保存索引与元信息
- 音效文件属于项目资源，不进入数据库

### 8.2 原因

- 预览资源可能较大，不适合直接塞进 `clipboard_items`
- 异步准备类资源需要独立状态与失败原因，放在单独表更清晰
- 资源缓存可复用，有助于减小重复解析成本

## 9. 分阶段落地建议

### 9.1 第一阶段（P0）

- 音效英文资产入库与播放服务
- 预览联动状态模型与窗口复用
- 音频 / 视频预览
- PDF 预览

### 9.2 第二阶段（P0 / P1）

- `DOCX / XLSX / PPTX` 结构化内容预览
- 链接摘要抓取与预览
- 预览资源缓存与状态事件补齐

### 9.3 第三阶段（P1）

- `DOC / XLS / PPT` 轻量抽取与降级兼容
- 预览失败分析日志与回归治理
- 性能优化与批量测试数据扩展

---

## 10. 最终结论

- `v1.5` 在当前架构下可以落地，且大部分能力属于 `v1.4` 现有基础上的增量扩展
- 应优先采用“`Howler.js` 音效播放 + `Vidstack` 媒体预览 + 单窗口预览联动 + 分层预览渲染 + 异步预览缓存”的组合方案
- 关键成功前提不是追求所有格式一步到位，而是**先把主链路打通，再为复杂格式提供可接受的结构化降级**
