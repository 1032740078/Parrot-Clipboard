# v1.5 UX 设计文档

## 1. 文档概述

### 1.1 版本范围

本文档只描述 `v1.5` 的体验设计目标：反馈更明确、预览不断流、媒体更可直接消费、文稿与链接更可理解。

### 1.2 设计原则（`v1.5` 强调）

- **反馈即时但克制**：复制、粘贴成功可保留声音反馈，预览默认保持安静，避免形成噪音轰炸
- **预览不断流**：打开预览后，用户切换卡片不应反复开关窗口
- **内容比文件名更重要**：能展示正文、摘要或结构化内容时，不应只停留在路径列表
- **降级可理解**：失败时要告诉用户“为什么看不到”，而不是只留下空白区域
- **保留主流程稳定**：新增预览能力不能破坏 `v1.4` 已有文本、图片、文件列表体验

---

## 2. 主面板 UX

### 2.1 信息架构目标

- 继续保持 `v1.4` 的搜索优先与类型筛选布局
- 不在主面板新增额外播放器或复杂详情区，避免首屏被挤压
- 主面板的新增体验重点是“预览联动”，而不是“把预览塞回面板”

### 2.2 预览联动规则

- 只有在用户显式打开预览后，主面板切换选中卡片才驱动预览联动
- 关闭预览后立即停止联动，避免用户只想浏览列表时被打扰
- 预览切换应该优先“平滑刷新内容”，而不是窗口闪烁或重开

### 2.3 音效触发规则

- 新记录成功入库：播放复制音效
- 粘贴成功完成：播放粘贴音效
- 预览首次显示成功：默认不播放预览音效，避免连续浏览时打扰用户
- 去重命中、失败、空结果与关闭动作默认不播放成功音效

---

## 3. 类型与卡片 UX

### 3.1 类型命名与预览预期

- `音频`：用户预期是“可直接听”
- `视频`：用户预期是“可直接看”
- `文稿`：用户预期是“能看到主要内容”
- `超链接`：用户预期是“知道这个链接指向什么”

### 3.2 卡片与预览的关系

- 卡片负责快速识别，不承担复杂内容消费
- 预览负责完整确认，不要求主面板卡片承载媒体播放器或文稿正文
- 当无法进入内容级预览时，至少保证卡片摘要与预览 fallback 信息一致

---

## 4. 预览窗口 UX

### 4.1 总体目标

- 让预览窗口成为连续浏览器，而不是一次性的详情弹窗
- 文本、图片、音频、视频、文稿、链接在同一窗口内共享统一外壳与状态反馈
- 不同渲染器可以不同，但 loading、错误、fallback 的语言必须一致

### 4.2 音频预览规则

- 默认聚焦播放能力，不做复杂音频编辑界面
- 至少展示播放 / 暂停、进度、时长与文件信息
- 失败时回退为文件名、路径与错误说明

### 4.3 视频预览规则

- 默认显示视频画面或封面图
- 控件应足够直观，避免用户必须猜测如何播放
- 当视频无法解码时，使用统一 fallback 卡片而不是黑屏

### 4.4 文稿预览规则

- PDF 预览优先显示正文页面
- `DOCX / XLSX / PPTX` 预览以“可读结构化内容”为优先目标
- 不要求像原生办公软件那样完全还原排版、批注与动画
- 旧格式解析失败时，要明确告诉用户这是“不支持或解析失败”，而不是假装内容为空

### 4.5 超链接预览规则

- 至少展示 URL、标题、摘要、站点信息
- 若有主图，可作为辅助信息，但不能压过文字信息
- 提供“在默认浏览器打开”入口，作为内容不完整时的自然补充路径

---

## 5. 设置页与关于页 UX

### 5.1 设置页

- `v1.5` 不新增音效与预览设置分组
- 维持 `v1.4` 的简洁工具风格，避免为本版能力临时塞入未经验证的开关项

### 5.2 关于页

- 保持 `v1.4` 的精简定位
- 不承载音效鸣谢、第三方预览引擎说明等大段内容，相关信息更适合放在开发文档或发布说明中

---

## 6. 动效与可访问性

### 6.1 动效

- 预览联动时使用轻量淡入或内容替换过渡即可
- 禁止每次切换目标记录都伴随明显窗口跳动或重建动画
- 文稿与链接加载态可以使用低干扰的骨架屏或文本 loading，不建议复杂转场

### 6.2 可访问性

- 音效不是唯一反馈来源，关键状态仍需可见文案
- 音频 / 视频控件要支持键盘访问
- 文稿与链接 fallback 需要可复制文件路径或 URL
- 错误态必须可读，不能只显示图标

---

## 7. 平台特定 UX 规范

### 7.1 macOS

- 保持 `v1.4` 的玻璃浮层风格
- 媒体控件与预览内容区注意与系统窗口拖拽区区分，避免误拖拽

### 7.2 Windows

- 媒体与文稿预览优先保证可读性与稳定性，不依赖只有 macOS 才有的视觉能力
- 若部分编码兼容性不足，应更明确展示降级说明

### 7.3 Linux

- 对不稳定的多媒体或文稿能力优先采用结构化降级
- 所有预览布局不能依赖单一系统特性才能成立
